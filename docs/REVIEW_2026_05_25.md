# Phase 0 v3.1 Architectural Review Summary (2026-05-25)

This document captures what changed between Phase 0 v3 and v3.1, and
why. Read this together with `docs/REVIEW_2026_05_09.md` (v2) and
`docs/REVIEW_2026_05_19.md` (v3) for the full lineage.

## Summary

Phase 1 implementation planning surfaced two categories of problem
that needed resolution before code starts:

1. **Document drift in `openapi.yaml`** that didn't catch up with v2's
   schema changes. The five-state liveness model, the v2 federation
   cert split, the severe-content moderation rule, the legal-hold
   flag, and the missing typed responses for integrity/possession
   audit listings would all have failed `oapi-codegen` against the
   live SQL schema at Phase 1 build time.
2. **Architectural-review feedback** that produced two roadmap
   changes and one Phase 1 implementation constraint. Specifically:
   `pkg/node` cannot honestly be semver-stable in Phase 1 before
   Phase 2 exercises it; and single-shot AEAD as currently spec'd
   restricts Nova's broader media mission to full-object fetch
   patterns incompatible with modern web playback, CDN partial-
   object caching, and reasonable TTFB on large objects. The
   streaming-AEAD envelope variant — previously listed under Phase
   6+ research — was promoted to a Phase 2 deliverable.

v3.1 also records small topology-hardening notes (admin/public
origin separation, ephemeral first-run wizard, file-based secret
injection) that the implementation plan will then honor.

## Findings and resolutions

### A. OpenAPI document drift (fixed)

| # | Drift | Resolution |
|---|---|---|
| A1 | `Node.status` enum in `openapi.yaml` still listed `[active, degraded, offline, revoked]` (v1 four-state). DATA_MODEL.sql and FEDERATION_PROTOCOL.md v2 are five-state `[active, suspect, unreachable, evicted, revoked]`. | Updated `openapi.yaml` `Node.status.enum` to the five-state set with an inline description pointing at FEDERATION_PROTOCOL.md § "Liveness state machine (v2)". |
| A2 | `Node` schema missing v2 fields `federation_cert_fingerprint`, `reputation_score`, `last_status_change_at`. | Added all three; marked `federation_cert_fingerprint` and `last_status_change_at` as `required`. The two-cert authentication model (Nebula cert + federation cert) is now correctly represented in the public API. |
| A3 | `ModerationDecision.rule` enum omitted `severe_content`, even though `DATA_MODEL.sql` and `SEVERE_CONTENT_PROCEDURE.md` use it. | Added `severe_content` to the enum with a description pointing at the legal-hold procedure. |
| A4 | `ModerationDecision` schema omitted `legal_hold` and `scheduled_tombstone_at` fields. The v2 quarantine-first DMCA flow and the severe-content preservation flow both depend on them. | Added both fields with descriptions tying them to `DMCA_PROCEDURE.md` § "Lifecycle (quarantine-first default)" and `SEVERE_CONTENT_PROCEDURE.md` § "Intended state machine". |
| A5 | `listIntegrityAudits` (`GET /api/v1/admin/audits/integrity`) had a `200` response with description only and no typed schema. Same for `listPossessionAudits`. | Added typed `IntegrityAudit`, `PossessionAudit`, `PaginatedIntegrityAudits`, `PaginatedPossessionAudits` schemas in `components/schemas` and bound them to the listing responses. Also typed the `audit_kind` query parameter against the seven Phase 1 audit kinds. |

### B. Roadmap and architectural changes (now resolved)

| # | Issue | Resolution |
|---|---|---|
| B1 | `ROADMAP.md` Phase 1 deliverable list says "Exports `pkg/coordinator` and `pkg/node` as semver-stable Go library packages." Phase 1 has no `cmd/node` binary; the donor binary ships in Phase 2. Freezing a public Go interface before any caller exercises it would produce immediate semver churn at Phase 2 (when reality almost certainly forces v2). | `pkg/node` deferred to Phase 2. Phase 1 keeps node-side types under `internal/node/`. `pkg/coordinator` ships, but its `product` subpackage is marked `v0.x.y` until Phase 4 when external adapter authors are real consumers. `ROADMAP.md` updated accordingly. |
| B2 | Streaming-AEAD listed under Phase 6+ research. Single-shot AEAD restricts encrypted blobs to full-object fetch: HTTP `Range` returns 416, CDN partial-object caching is defeated, TTFB on large objects requires authenticating the entire ciphertext before any plaintext byte streams. For a project whose mission includes `nova-video`, `nova-audio`, large `nova-archive` objects, and modern web media expectations, deferring this to Phase 6+ effectively constrains Nova to a thumbnail/profile-image host until then. | Promoted to a Phase 2 deliverable alongside federation. Federation is the right pairing because per-block crypto semantics share infrastructure with possession audits and donor-to-donor repair: chunk boundaries align with the existing 256 KiB IPFS block boundary, so chunk N == block N. `ROADMAP.md` Phase 2 amended; Phase 6+ entry removed. `ENCRYPTION_ENVELOPE.md` gains a § "Planned v2: Streaming-AEAD" with the wire-format sketch and the Phase 1 implementation constraints (versioned `Codec` interface, `envelope_version` in JSON metadata, `X-Nova-Envelope-Version` response header). `ARCHITECTURE_DECISIONS.md` T1.2 reframed around the versioned wire format; T1.7a added for the v2 codec. |

### C. Topology hardening (folded into specs)

| # | Hardening | Resolution |
|---|---|---|
| C1 | Public-content origin and admin/setup origin should be distinct virtual hosts so XSS or cache poisoning at the public origin doesn't become administrative trust. Defense-in-depth, no protocol change. | `THREAT_MODEL.md` trust-boundary ① note added. The Phase 1 design will ship two nginx server blocks. |
| C2 | First-run setup wizard should be ephemeral: loopback-only by default, disabled permanently after first successful bootstrap, refuses to complete until the operator confirms master-key backup. | `THREAT_MODEL.md` adds boundary ① a covering wizard ephemerality. Phase 1 design will implement the `.bootstrap-complete` sentinel and forced-readback master-key confirmation. |
| C3 | Master key loading should support both env vars and file mounts (Docker secrets, Kubernetes secret mounts) without favoring one. | `THREAT_MODEL.md` boundary ③ updated to "environment variable OR file-mount". Phase 1 implements the `NOVA_FOO` → `NOVA_FOO_FILE` → `/run/secrets/foo` resolver chain. |

## What did not change

- The donor-blind storage model.
- The single-coordinator topology and the out-of-scope-ness of
  multi-master HA. **The project's stated trust model is preserved.**
  The architectural review flagged this; the project framing
  affirms it.
- The Nebula private-mesh authentication model and v2 mTLS-over-
  Nebula application layer.
- The 256 KiB deterministic IPFS chunker.
- The five-state liveness model, mass-casualty and slow-attrition
  detection, and the Tier 1 strict-priority healing rule.
- The quarantine-first DMCA flow and the legal-hold preservation
  mechanism.
- The license (Apache-2.0).

## What still needs deliberation before Phase 2

The v2 streaming-AEAD wire-format sketch in
`ENCRYPTION_ENVELOPE.md` § "Planned v2: Streaming-AEAD" is a design
sketch, not the authoritative spec. Phase 2 implementation begins
with that section being promoted to a full normative spec with:

- Authoritative test vectors covering chunk-boundary edges, the
  final-chunk marker, and AAD substitution attacks.
- Resolution of the AAD-with-CID chicken-and-egg via an
  intermediate commitment scheme.
- Per-block fetch semantics under federation (does the orchestrator
  push individual blocks for partial reads, or always whole
  envelopes; if the former, how does it interact with the existing
  repair-token-per-CID model).
- Range-request error semantics for partially-corrupt envelopes.

These are not v3.1 work. They become v4 work when Phase 2
implementation starts.
