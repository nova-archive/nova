# Phase 0 v2 Design Review Summary (2026-05-09)

This document captures what changed between the original Phase 0
specs and Phase 0 v2, and why. Future contributors who wonder why a
spec looks the way it does should read this first.

## Summary

The original Phase 0 specs were completed and committed across seven
commits ending with the roadmap mark-complete on 2026-05-09. A
design-audit pass identified five hard-correctness bugs, several
underspecified architectural choices, and one major cross-document
drift problem. Five additional commits the same day landed the v2
revisions.

The implementation gate moved from the original Phase 0 to Phase 0
v2: no production code is allowed to lock in against the v1 specs;
Phase 1 implementation begins from the v2 surface only.

## Findings and resolutions

### A. Hard-correctness bugs (now fixed)

| # | Bug | Resolution |
|---|---|---|
| A1 | Signed-URL revocation prefix scheme could not match `cid:`/`aud:`/`kid:` against canonical strings starting with the URL path | Replaced with structured revocation: `signed_url_revocations(kind, value)` with verifier parsing canonical fields |
| A2 | OpenAPI promised `Range` on `/blob/{cid}` but envelope is single-shot AEAD | Range now supported only on `public_archival` plaintext blobs; encrypted blobs return 416 to Range. Streaming AEAD deferred to Phase 6+ |
| A3 | `OnUpload` documented as running both before *and* after CID computation in the same spec | Hook split: `AnalyzeUpload(plaintext)` runs on plaintext before encryption; `OnCommitted(blob, metadata)` runs after blob row commit |
| A4 | Derivatives lived in a separate `derivatives` table without their own encryption keys, state, or moderation cascade | Derivatives are now first-class `blobs` rows with `parent_cid`, own keys, own state; tombstone cascades from parent |
| A5 | `public_archival` referenced in specs but no column existed | Added `collections.public_archival boolean` with CHECK constraint requiring `visibility = 'public'` |
| A6 | `expected_cid` upload field assumed client could compute the envelope CID, but random nonces make that impossible | Removed for Phase 1; reconsider with deliberate plaintext-hash dedup design in Phase 3 |
| A7 | `keys` table conflated per-blob data keys (per-upload lifecycle, shred) with HMAC signing keys (rotated, grace window) | Split into `data_encryption_keys`, `signing_keys`, `master_key_versions` |

### B. Architectural choices needing explicit decision (now resolved)

| # | Choice | Resolution |
|---|---|---|
| B1 | CDN cache policy was uniform `public, immutable` for all `/i/*` and `/blob/*` | Stratified by collection visibility + blob state + auth class. Private/signed are `private, max-age=300, must-revalidate`; quarantined/tombstoned are `no-store`. CDN reframed as optional with explicit decryption-on-ingress warning. |
| B2 | Federation auth conflated Nebula overlay identity with HTTP application identity | HTTPS+mTLS inside Nebula with a separate federation client cert. Both certs issued by operator CA; coordinator's `/fed/v1` requires both layers |
| B3 | Healing source selection used generic `ipfs pin add` (Bitswap chooses the source) | Donor-to-donor controlled repair: orchestrator embeds source designation + HMAC repair token in the change-log entry; donor fetches from designated source via `/fed/v1/blob/{cid}` over Nebula. Bitswap-backed pin add is explicitly disabled for repair |
| B4 | Liveness model collapsed "missed heartbeat" with "long-term gone" into a 30-day timer | Five-state model: `active`, `suspect`, `unreachable`, `evicted`, `revoked`, with separate timers. Healing engages at `unreachable` (~1h), not at `evicted` (30d) |
| B5 | Default replication factor 3 produces 6.4 % data loss in the 40-% mass-casualty scenario | R is now operator-configurable per content class (`important` defaults 3, `normal` 3, `cache` 2). Operators with sufficient donor capacity should raise `important` to 5 for archival-grade durability. No code refactor required to change R |
| B6 | DMCA action immediately tombstoned and crypto-shredded before counter-notification window | Quarantine-first is the new default: state to `quarantined`, signed-URL revocation, scheduled tombstone after the counter-notification window. Operators can opt into immediate-tombstone via config |
| B7 | Master-key rotation deferred to Phase 5 | Promoted to Phase 1: `novactl keys rotate-master`, online rotation via `master_key_version_id` on each wrapped key. A deployment that cannot rotate its master key is not safe |
| B8 | Severe-content (CSAM-class) handling not separated from DMCA, risking destruction of evidence required by 18 U.S.C. § 2258A | Schema gates crypto-shred via `data_encryption_keys.legal_hold`; quarantine + legal_hold preserves bytes and key for the statutory period; full automated workflow deferred to Phase 4 with manual Phase 1 path documented |

| # | Spec | Phase | Purpose |
|---|---|---|---|
| C1 | `docs/specs/IPFS_IMPORT_RULES.md` | Phase 0 | Deterministic IPFS import parameters; CIDs are bit-identical across implementations. Prerequisite for proof-readiness |

### D. Polish edits (claims narrowed, language tightened)

- Crypto-shred GDPR Article 17 claim narrowed: shredding is one
  component of an erasure procedure, not the whole thing.
  Operators are responsible for CDN/cache/backup/derivative
  cleanup beyond the per-blob key destruction.
- `auth: anonymous` mode for management endpoints removed from
  production builds; available only behind `nova_dev` build tag
  with loopback binding.
- `perceptual_hash` removed from public Image schema (cross-site
  fingerprinting risk); operator API only.
- `THREAT_MODEL.md` now explicitly frames Nova as donor-blind, NOT
  operator-blind. The trust model is "pick an operator you trust,
  or run your own"; the operator sees plaintext on every read and
  on transform. This was implicit before and is now explicit.
- `OPERATOR_CHECKLIST.md` [REQUIRED] reduced to four items the
  coordinator literally cannot run safely without. Everything else
  moved to [RECOMMENDED] with operator judgment framing.
- `README.md` adds explicit "no third-party traffic intermediation
  by default" design priority and trust-model note.

### Cross-document drift (fixed)

The most subtle finding from the third design review was
cross-document drift: `DATA_MODEL.sql` had been updated to v2 but
`DMCA_PROCEDURE.md`, `OPERATOR_CHECKLIST.md`,
`FEDERATION_PROTOCOL.md`, `HEALING_PROTOCOL.md`, and
`PRODUCT_MODULE_INTERFACE.md` still referenced the old `keys`
table, prefix-based revocation, four-state liveness, and the
OnUpload before/after-CID contradiction.

The v2 consistency pass rewrote those five documents in full
rather than spot-editing, because the changes were pervasive
enough that surgical edits would have introduced new
inconsistencies.

## What did not change

These items were correct in v1 and remain unchanged:

- The donor-blind storage model (encryption-by-default).
- The single-coordinator topology and explicit out-of-scope-ness
  of multi-coordinator HA.
- The Nebula private-mesh authentication model at the overlay
  layer.
- The `KUBO_HARDENING.md` validator rules and refuse-to-start
  semantics.
- The `simulations/orchestrator_resilience.py` empirical findings.
- The `THREAT_MODEL.md` scope (the trust-boundary diagram is
  unchanged; only the framing language gained explicit
  donor-blind-not-operator-blind clarification).
- The license (Apache-2.0).

## What still needs deliberation before Phase 2

The v2 federation specs commit Nova to mTLS-inside-Nebula plus
donor-to-donor repair transport. Phase 2 implementation will
exercise these designs against real donor variability and may
surface implementation-level issues the specs do not anticipate.
The v2 specs are the contract Phase 2 implements faithfully;
divergences require a v3 amendment.

## What still needs deliberation before Phase 4

The severe-content procedure (`SEVERE_CONTENT_PROCEDURE.md`) is a
Phase 4 deliverable. Operators of public-upload Phase 1
deployments must follow the manual path documented there.
Operators considering large-scale public deployments should treat
Phase 4's automated workflow as a hard prerequisite for opening
public uploads to anonymous users.

## Implementation gate for Phase 1

Phase 1 implementation may begin against the v2 specs. The five
commits above represent the complete revision; subsequent spec
changes go through normal contribution review with explicit
amendment notes pointing back to this document.
