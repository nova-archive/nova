# Phase 0 v3 Design Review Summary (2026-05-19)

This document captures what changed between Phase 0 v2 and v3, and
why. Future contributors who wonder why a spec looks the way it does
should read this and `docs/REVIEW_2026_05_09.md` together.

## Summary

Phase 0 v2 was the result of a design-audit pass that fixed five
hard-correctness bugs and resolved several underspecified
architectural choices. v3 follows a second round of architectural
review focused on three questions:

1. What is the architectural-decision boundary between protocol-
   enforced commitments, operator-tunable parameters, and operator
   freedoms? v2 specs documented individual decisions but no
   document drew the line between them.
2. What did v2 leave drifting? In particular, `THREAT_MODEL.md`'s
   trust-boundary diagram still referenced libp2p as the donor-to-
   donor transport even though v2 replaced that with controlled
   HTTPS-over-Nebula repair fetches.
3. What failure modes does the existing healing math miss? Mass-
   casualty detection catches burst failures; the slow-attrition
   case (gradual donor loss crossing the threshold for sustainable
   replication) had no early-warning signal.

v3 also documents a small number of recurring "but have you
considered X?" architectural asks (threshold cryptography, end-to-
end encryption, PSI/ZKP moderation, S3 API, multi-master HA) with
explicit rationale tied to design principles, so the project can
point to a written answer instead of having the conversation again.

## Findings and resolutions

### A. Document drift from v2 transition (fixed)

| # | Drift | Resolution |
|---|---|---|
| A1 | `THREAT_MODEL.md` boundary ⑤ in the trust-boundary diagram and table referenced "libp2p inside private swarm" as the donor-to-donor transport, contradicting `FEDERATION_PROTOCOL.md` v2's controlled HTTPS-over-Nebula repair transport | Updated to "controlled HTTPS over Nebula + repair tokens"; the table entry now correctly describes mTLS + coordinator-issued repair tokens, with the IPFS swarm key noted as defense-in-depth gating Kubo's daemon-level peering rather than as the donor↔donor data channel |
| A2 | `THREAT_MODEL.md` § A "Malicious donor node" capabilities text said donors "receive ciphertext blobs through the libp2p swarm" | Rewritten to describe the federation client cert, polling `/fed/v1/pins/changes`, and fetching blobs over the controlled HTTPS-over-Nebula repair transport with valid repair tokens. Added an explicit mitigation point that donors cannot mint repair tokens or ad-hoc-fetch blocks |
| A3 | `IPFS_IMPORT_RULES.md` referred to the UnixFS-1 maximum link count of 174 as the "libp2p default," which is misleading — the parameter is a Kubo UnixFS-1 default, not a libp2p one | Changed to "Kubo's UnixFS-1 default" |
| A4 | `IPFS_IMPORT_RULES.md` § "What MAY vary" said transport-level details "is libp2p's domain"; Nova no longer uses libp2p for federation transport | Changed to point at the federation protocol's "Repair transport" section, which is the relevant domain in v2+ |

### B. Missing classification (now added)

| # | Gap | Resolution |
|---|---|---|
| B1 | No single document classified architectural decisions by amendment authority (protocol-enforced vs operator-tunable vs operator freedom) | New `docs/specs/ARCHITECTURE_DECISIONS.md` with the three-tier classification table and cross-references to each spec where the decision lives |
| B2 | The "trust-model choices not implemented" rationale (threshold cryptography, end-to-end encryption, PSI/ZKP moderation, S3 API) was distributed across out-of-scope sections and the README's design priorities, requiring readers to assemble it themselves | New "Trust-model choices not implemented" section in `THREAT_MODEL.md` consolidates the rationale, tying each "we don't do this" to the design principles it would conflict with |

### C. Missing failure-mode coverage (now added)

| # | Gap | Resolution |
|---|---|---|
| C1 | Mass-casualty detection only fires on burst failures (>20% loss within 1 hour, configurable); steady attrition crossing the math threshold for sustainable replication had no early-warning signal | New "Slow-attrition detection" section in `HEALING_PROTOCOL.md` with the `federation.shrinking` webhook, the `capacity_runway_days` metric, and the `capacity_runway_floor_days` configuration parameter (default 7) |
| C2 | "Acknowledged residual risks" listed traffic-volume analysis and silent-coordinator compromise, but had no entry for federation slow-attrition collapse | New residual-risk item #7 in `THREAT_MODEL.md` describing the slow-attrition mode and naming the `federation.shrinking` webhook as the early-warning signal |
| C3 | The orchestrator resilience simulation modeled only burst failures; sybil concentration and slow-attrition scenarios were not validated against the architecture's invariants | Three new simulations: `sybil_concentration.py`, `long_tail_churn.py`, `key_rotation_load.py` |

### D. Operator-side patterns (new recipes)

The architecture supports several patterns the protocol does not
itself ship. v3 documents three of them as recipes so operators
have a written reference and so contributors can point to "this is
operator-side, not protocol-side" when the feature gets requested.

| # | Recipe | What it solves |
|---|---|---|
| D1 | `docs/recipes/AUTOMATED_ONBOARDING.md` | Self-serve donor enrollment via sub-CA + friction layer + quarantine subnet, building on the existing cert issuance/revocation API |
| D2 | `docs/recipes/KEY_ESCROW.md` | Shamir-split backup escrow for `NOVA_MASTER_KEY` versions, preserving operator sovereignty for the read path |
| D3 | `docs/recipes/COLD_STANDBY.md` | Manual-failover hot-spare host for read-availability after primary failure, without introducing multi-master complexity |

### E. Roadmap clarifications

`Phase 6+ — Research` now explicitly names several speculative
directions that recur in architectural discussions: read-only
secondary coordinator, streaming-AEAD envelope variant, formal PDP/
PoR, hot/cold tier auto-migration, and the optional S3 read-only
adapter product layer.

## Cross-document drift (fixed)

The v3 changes are spread across four files but introduce no
cross-document drift because each change is either:

- Additive (new section in an existing spec, new file in `docs/`),
- Removing a contradiction with v2's federation/healing specs (the
  libp2p references in `THREAT_MODEL.md` and `IPFS_IMPORT_RULES.md`),
  or
- A classification document that points at existing specs rather
  than restating their content.

The v3 commits in chronological order:

1. `docs/specs/ARCHITECTURE_DECISIONS.md` (new)
2. `docs/THREAT_MODEL.md` (libp2p drift fix + new trust-model
   choices section + new residual risk)
3. `docs/specs/HEALING_PROTOCOL.md` (slow-attrition detection)
4. `docs/specs/IPFS_IMPORT_RULES.md` (libp2p touch-up)
5. `docs/recipes/` (three new files + README index update)
6. `simulations/` (three new files + README update)
7. `docs/ROADMAP.md` update + this review summary

## What did not change

These items were correct in v2 and remain unchanged:

- The donor-blind storage model (encryption by default, donors
  hold ciphertext only).
- The single-coordinator topology and the explicit out-of-scope-
  ness of multi-master HA.
- The Nebula private-mesh authentication model at the overlay
  layer; the v2-introduced mTLS-over-Nebula application layer.
- The donor-to-donor controlled repair transport with HMAC-signed
  source-and-destination-pinned tokens.
- The `KUBO_HARDENING.md` validator rules and refuse-to-start
  semantics. The remaining libp2p references in
  `KUBO_HARDENING.md` are intentional: they describe Kubo's
  internal libp2p stack being constrained by the private swarm
  key, which is *the rationale* for not relying on libp2p for
  federation transport.
- The `simulations/orchestrator_resilience.py` empirical findings.
- The five-state liveness model and the 20%-in-1-hour mass-casualty
  detection.
- The crypto-shred + legal-hold gating model.
- The `DMCA_PROCEDURE.md` quarantine-first default and Phase 4
  severe-content workflow.
- The license (Apache-2.0).

## What still needs deliberation before Phase 1

The Phase 1 implementation gate moves from v2 to v3. No production
code is allowed to lock in against pre-v3 specs. The five v2
commits + the v3 commits above represent the complete revision.

Phase 1 will exercise the integrity-audit, master-key-rotation, and
deterministic-IPFS-import surfaces against real implementations.
Specifications that turn out to be wrong or underspecified at that
stage will produce a v4 amendment with the same pattern: targeted
fixes, cross-document consistency pass, a review summary, and a new
implementation gate.

## What still needs deliberation before Phase 2

The `federation.shrinking` webhook's threshold defaults
(`capacity_runway_floor_days = 7`) were chosen from the
`long_tail_churn.py` simulation's warning-lead-time analysis. Phase
2 implementation should re-validate against real donor variability;
the threshold may need to move based on observed attrition patterns
in the field. Specifically, federations with very slow attrition
(< 1 % per week) may prefer a higher floor to give recruitment more
time; federations with bursty churn may need to tune the
suppression window in concert with `mass_casualty_window_seconds`.

The recipes (`AUTOMATED_ONBOARDING`, `KEY_ESCROW`, `COLD_STANDBY`)
are not implementation deliverables; they are reference patterns.
The Phase 2 implementation does not include them. Operators choose
to build them or not, on their own infrastructure.

## Implementation gate for Phase 1

Phase 1 implementation may begin against the v3 specs. The commits
listed above represent the complete revision; subsequent spec
changes go through normal contribution review with explicit
amendment notes pointing back to this document.
