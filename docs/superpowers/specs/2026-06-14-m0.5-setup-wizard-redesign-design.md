# P2-M0.5 — Setup-wizard redesign (`web/setup`)

## Context

The first-run setup wizard shipped in M13 (`web/setup`, driving the
sentinel-gated `/setup/*` coordinator seam). Bug's first-run friction review of
v0.1.0-rc1 and the M0.2 privacy-posture work both flagged the same gap: the
wizard never *explains* its load-bearing choices at the moment the operator makes
them. `paranoid` is an opaque single toggle — "Hardened defaults for hostile
environments" — that says nothing about *what* it changes, and security/crypto
jargon (master key, fingerprint, Certificate Transparency, IPFS DHT, ToS) is
presented with no plain-language help. A new operator can't give informed consent
where it matters.

M0.2 already reframed the **backend**: `paranoid` is a default-off *preset* over
individually-addressable constituents — outbound `webhooks`,
`source_ip_retention_days`, `coordinator.record_source_ip` (tri-state `*bool`),
`coordinator.public_ipfs_dht` — with **warn-not-force** semantics
(`internal/config/paranoid.go: ApplyPrivacyPreset`): the preset sets protective
defaults, an explicit operator value wins, and relaxing one only *warns*. M0.5
builds the **wizard UX** on that model. It is the last operator-onboarding
remediation before M0.6 (the admin Settings modal, which owns granular runtime
tuning via the M0.4 config API).

**Scope.** A `web/setup` redesign around three pillars — consequence copy,
jargon info-buttons, paranoid delineation — plus a small additive
`setup.Answers` + `render.go` expansion so the new per-setting toggles actually
reach `operator.yaml`. **Non-goals:** no wiring to the M0.4 runtime config API
(that is M0.6); no surfacing of every `operator.yaml` field (granular tuning is
M0.6); no visual rebrand (the wizard keeps its M13 brand-tokened CSS). Depends on
M0.2.

## Decisions (ratified with Bug, 2026-06-14)

- **Paranoid is a tri-state parent toggle** over individually exposed child
  toggles. Checking the parent checks every child; unchecking any child drops the
  parent to **indeterminate (grey)**; checking the parent from
  indeterminate/empty re-checks every child. This is the familiar "select-all"
  disclosure, applied to privacy hardening.
- **Child set = three live toggles + two read-only info rows.** Live: don't
  record uploader IPs, keep IP logs 1 day (not 30), keep pinned CIDs off the
  public DHT. Info-only: no outbound webhooks (none exist at first run), metrics
  stay loopback-only (no `operator.yaml` field; startup-enforced). These are
  exactly the constituents `ApplyPrivacyPreset` touches, minus the two that
  aren't first-run-tunable.
- **Jargon help is a native `<details>`/`<summary>` disclosure**, not a
  positioned popover — zero JS positioning, can't overflow the card, accessible
  by default. Made screen-reader-friendly with a labelled summary.
- **Step flow folds the content-free `keys` step** into a one-line reassurance on
  the master-key step, and renames `tos`→"Public uploads",
  `paranoid`→"Privacy & hardening". External-IdP auth stays hidden at first run
  (local-only); it is an advanced/post-setup concern.

## The three pillars

### A — Consequence copy
Every load-bearing choice states its consequence *at decision time*, mirroring
the `ApplyPrivacyPreset` warning text so the operator reads it in the wizard, not
only later in a startup log. Touch points: TLS modes (per-mode privacy/cost — the
http-01 Certificate-Transparency warning exists today; extend the other modes),
public uploads (the T1.20 ToS requirement + the abuse-surface it opens), the
three paranoid constituents, and the master key (already strong — left as is).

### B — Jargon info-buttons: "learn-this" vs "abstract-away"
A single glossary module (`wizard/glossary.ts`) maps each term to
`{treatment, label, body}`. Two treatments:

- **learn-this** — the term renders with a dotted underline followed by a ⓘ
  disclosure that expands a 1–2 sentence plain-English explanation. For decisions
  the operator must understand to run the node safely: *master key, fingerprint,
  bootstrap token, TLS mode / Certificate Transparency, Terms of Service (T1.20),
  public uploads, source-IP recording, public IPFS DHT, paranoid.*
- **abstract-away** — a muted "handled automatically" treatment with **no** info
  button, for internal cryptographic machinery the operator can't and shouldn't
  tune: *swarm identity key, content-signing key, sealing / keystore, replication
  factor.* This is precisely what the standalone `keys` step is today; it folds
  into one reassuring line on the master-key step.

The taxonomy rule: **learn-this** = an operator-facing decision or a
recovery-critical fact; **abstract-away** = machinery with no operator lever.

Accessibility: the `<summary>` carries the accessible name
(`aria-label="What is a master key?"`) and the ⓘ glyph is `aria-hidden`. Native
`<details>` already exposes the expanded/collapsed state to assistive tech, so no
manual `role` / `aria-expanded` wiring is added (or wanted).

### C — Paranoid delineation (tri-state parent + children)

```
[▣] Harden privacy (paranoid)                          ⓘ   ← tri-state parent
     Sets every protection below. Uncheck any one to relax it.
 ├ [✓] Don't record uploader IP addresses              ⓘ   live toggle
 │      Off → each blob stores the client's source IP.
 ├ [✓] Keep IP logs 1 day, not 30                      ⓘ   live toggle
 │      Only applies while IP recording is on.
 ├ [✓] Keep pinned CIDs off the public IPFS DHT        ⓘ   live toggle
 │      Off → this node advertises stored CIDs publicly.
 ├ ·  No outbound webhooks         (none configured)        info-only row
 └ ·  Metrics stay loopback-only   (not internet-exposed)   info-only row

Indeterminate (one child relaxed):
 [▦] Harden privacy (paranoid) — partial               ⓘ
 ├ [ ] Keep IP logs 1 day, not 30        ⚠ 30-day IP logs
```

Each child stores its **protective** state (checked = hardened); all init
unchecked, preserving today's default-off posture. The parent's three states are
derived from the children (all on → checked; all off → unchecked; mixed →
indeterminate). Clicking the parent: checked → set all children off; otherwise →
set all children on.

### Wire mapping (WYSIWYG `operator.yaml`)
On commit, `toAnswers` emits **explicit** constituent values so the file mirrors
exactly what the operator saw:

| Form (protective state) | Answers field | Value |
|---|---|---|
| `hardenNoIPRecording` | `record_source_ip` | `!hardenNoIPRecording` |
| `hardenShortRetention` | `source_ip_retention_days` | `hardenShortRetention ? 1 : 30` |
| `hardenPrivateDHT` | `public_ipfs_dht` | `!hardenPrivateDHT` |
| (derived) | `paranoid` | `hardenNoIPRecording && hardenShortRetention && hardenPrivateDHT` |

`paranoid` is `true` only when the parent is fully checked; an indeterminate or
empty parent commits `paranoid:false` with explicit child values. The
indeterminate state is purely a UI affordance — it never persists as a
contradictory config.

**No wizard output ever trips an `ApplyPrivacyPreset` startup warning**, verified
against each warn branch: webhooks are empty at first run; whenever `paranoid` is
`true` the parent is fully checked, so `record_source_ip=false`,
`source_ip_retention_days=1`, and `public_ipfs_dht=false` — none of the warn
conditions (`record==true`, `retention>1`, `dht==true`, `len(webhooks)>0`) can
fire. The warn path stays reserved for its M0.2 purpose: hand-edited drift.

## Backend expansion (small, additive, backward-compatible)

- `internal/setup/answers.go` — add to `Answers`:
  `RecordSourceIP *bool`, `SourceIPRetentionDays int`, `PublicIPFSDHT bool`, each
  with **both** `json` and `yaml` `,omitempty` tags
  (`record_source_ip` / `source_ip_retention_days` / `public_ipfs_dht`). The yaml
  tags matter: `novactl setup` parses a headless answers file via
  `yaml.Unmarshal` (`cmd/novactl/main.go: loadAnswersFile`), so the new fields
  need real yaml keys for CLI parity with the web wizard. Add a friendly
  `Validate` bound (`source_ip_retention_days >= 0`); the authoritative floors
  still re-run when `render.go` round-trips the rendered YAML through
  `config.LoadFromBytes`.
- `internal/setup/render.go: RenderOperatorYAML` — map the three into
  `cfg.Coordinator.RecordSourceIP`, `cfg.SourceIPRetentionDays`, and
  `cfg.Coordinator.PublicIpfsDht` (replacing the hardcoded `PublicIpfsDht:
  false`).

No new endpoints: `/setup/answers` and `/setup/commit` already accept the
`Answers` JSON. The fields are `omitempty`, so a caller that omits them
(`novactl setup`, older clients) gets today's behavior — `nil`/`0`/`false`
flow into `ApplyPrivacyPreset`, which fills them from the preset exactly as now.

## Frontend structure

`web/setup/src/wizard/Wizard.tsx` is ~580 lines and will outgrow a single file
once info-buttons and the paranoid panel land. Extract the genuinely new, shared
pieces; keep the stepper shell and the simple step bodies in `Wizard.tsx` (itself
slimmed by the keys fold):

- **New:** `wizard/glossary.ts` (the term table), `wizard/InfoTerm.tsx` (the
  `<details>` learn-this term), `wizard/ConsequenceNote.tsx` (the consequence /
  warning line), `wizard/ParanoidSection.tsx` (the tri-state parent + children +
  info rows).
- **`wizard/types.ts`:** extend `FormState` with `hardenNoIPRecording`,
  `hardenShortRetention`, `hardenPrivateDHT` (all init `false`); rewrite
  `toAnswers` per the wire-mapping table (note the don't-record / keep-private
  polarity inversions).
- **`wizard/wizard.module.css`:** `.infoTerm` / `.infoBody`, child tree
  connectors, `.indeterminate`, `.consequence`, `.infoRow`.

The post-fold step order (10): `token`, `welcome`, `master-key`, `admin`, `tls`,
`public-uploads`, `privacy`, `review`, `commit`, `orientation`.

## Security / privacy notes

The redesign is informational and additive — it changes copy and surfaces
existing knobs; it adds no new secret, endpoint, or trust boundary. The bootstrap
token, master-key readback gate, and sentinel-gated `/setup/*` ephemerality
(THREAT_MODEL boundaries ①/①a) are untouched. T1.20 (public uploads → ToS) stays
a hard gate enforced by `Answers.Validate` and re-checked by the
`LoadFromBytes` round-trip. The new toggles can only move privacy settings within
the range M0.2 already sanctions, and the wizard now surfaces the same
consequence text the coordinator would otherwise only log — strictly more
informed consent, not less.
