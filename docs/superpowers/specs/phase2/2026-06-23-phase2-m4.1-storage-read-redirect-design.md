> **M4 proved that an acked donor holder is real; M4.1 is the milestone that lets
> the coordinator safely trust those holders for reads and pruning, without
> changing Nova's donor-blind, coordinator-served trust model.**

## Context

Phase 2 makes Nova's corpus durable across volunteer-run **donor nodes**. The
control plane (P2-M2 identity/registration, P2-M3 assignment sync) and the first
data plane (P2-M4 replication slice) are complete and merged:

- **P2-M4** (`p2-m4-replication-slice`) made an `ack` mean something real:
  coordinator-as-source signed grants, the donor
  fetch→deterministic-re-import→root-CID-verify→pin→persist→ack loop, donor
  `storage_max_bytes` + size enforcement, and crash-safe verified-holder state.
  Every later subsystem trusts the **acked-holder set as durability ground
  truth**. M4 deliberately built the replication **mechanism and nothing
  downstream of a verified holder** (D-M4-1): no donor-backed reads, no upload
  quorum, no origin pruning, no `coordinator_storage_mode`.

**P2-M4.1 is the storage/read redirect.** It carries the P2-M2.1 amendment M4
deferred and is **required before P2-M5**. It is the point where Nova stops
treating the coordinator's local Kubo as the canonical origin and starts treating
the **verified donor replica set as the durable substrate**: the coordinator
keeps metadata, keys, upload staging, and a *bounded* hot cache; cache misses are
donor-backed. The P2-M2.1 storage/read target is **not fulfilled until M4.1
lands**, and **P2-M5 must be designed assuming the coordinator origin copy is not
guaranteed.**

**The donor-blind invariant is preserved, not relaxed (T1.26).** M4.1 removes the
*coordinator-local-origin* requirement; it does **not** remove the coordinator
from the read path. The coordinator stays the **sole decrypt/serve point for
every byte**. On a cache miss it sources *ciphertext* from a donor, verifies it
by deterministic re-import **before** decrypting, decrypts, and serves. Donors
serve only ciphertext, and in M4.1 they serve it **only to the authenticated
coordinator** — never to end users, never (yet) to other donors.

## Decisions (ratified with Bug, 2026-06-23)

### D-M4.1-1 — Scope is the storage/read redirect, as one internally-staged milestone.

M4.1 implements exactly the P2-M2.1 read model and nothing M5 owns. It is **one
milestone** (one design, one plan), staged into independently shippable/testable
slices, **not** split into M4.1/M4.2: splitting read-redirect from
quorum/pruning would leave a half-transition in which donor-backed fetch exists
but the durability/visibility semantics that make M5's "no guaranteed origin"
assumption valid do not. The coordinator remains the sole decrypt/serve authority
throughout.

### D-M4.1-2 — A distinct coordinator federation **client** identity; do not overload the source-id constant.

Today federation peer identity is donor-shaped: `transport.IdentityFromCert`
parses **only** `nova://node/<uuid>` URI SANs and returns a `NodeID` with no role
concept. For a donor to authenticate the coordinator at its read-source endpoint,
the coordinator needs a real **client identity**, not a fake `nodes` row and not
an overloaded constant.

- Mint a coordinator federation **client** certificate whose URI SAN carries a
  distinct role: `nova://coordinator/<uuid>` (or a fixed `nova://coordinator/self`).
  `novactl` gains the issue path; the key loads via the existing
  `secret.ResolveSecret` chain.
- Extend the shared `transport` identity parser to return a **role** (`node` vs
  `coordinator`) alongside the id, so a verifier can branch on peer role. The
  parser stays pure-stdlib (donor-boundary-safe).
- `wire.CoordinatorSourceID` (`tokens.ReservedCoordinatorSourceID`) stays a
  **token-claim** source/dest constant only — it is **not** an mTLS identity and
  **not** a `nodes` row. The donor read-source handler accepts only the
  `coordinator` mTLS role and **never** looks the caller up in `nodes`.

This keeps the identity model honest: donor certs identify donors; the
coordinator server cert identifies the server; the coordinator **client** cert
identifies the coordinator as an authorized caller to donor read-source
endpoints.

### D-M4.1-3 — Minimal coordinator-only donor read-source endpoint, with strict assignment-bound verification.

A donor exposes `GET /fed/v1/blob/{cid}` on its Nebula-bound mTLS listener (the
inbound federation server is net-new in M4.1 — today `cmd/node` serves only
`/health`). It is the **only** new serving surface, and it is callable **only by
the coordinator client identity**. It is *not* the M5 donor↔donor repair server.

Possession alone is insufficient — a donor may still hold blocks from a stale or
unpinned assignment, an orphan, or an unpin/serve race. The token already carries
`assignment_id` + `generation` (`wire.Claims`); M4's coordinator endpoint never
checked them, M4.1's donor endpoint **must**. Before serving, the donor verifies:

```
caller mTLS role == coordinator                          (D-M4.1-2)
wire.Verify(coordinatorRepairPubKey, token, now) ok      (M4 pubkey, delivered via heartbeat)
token.source_node_id == this donor's node id             (donor is the SOURCE now)
token.dest_node_id   == reserved coordinator source id   (coordinator is the DEST)
token.cid            == {cid} path value
token.assignment_id / generation == local verified, acked-delivered assignment
token.not_before >= this donor's source_boot_time        (restart replay floor)
token.jti unused                                          (single-use replay cache)
ipfsclient.Has(cid) == true                              (recursively pinned, not block residue)
envelope_size <= token.max_bytes                         (else refuse; never truncate — D-M4.1-16)
egress budget: Bucket.Take(envelope_size, now) ok        (else budget_exceeded — D-M4.1-5)
```

Note the **reversed token direction** vs M4: in M4 the coordinator is the source
and the donor the dest; for a donor-backed read the donor is the source and the
coordinator the dest. The coordinator mints these grants (it holds the private
key); the donor only ever verifies (it holds the public key from M4's heartbeat).
A failure at any check is a clean refusal (`source_unauthorized` /
`blob_unavailable` / `budget_exceeded`) so the coordinator can try another holder.

### D-M4.1-4 — Extract donor-safe replay + token-source helpers; never import coordinator internals into the donor.

M4's replay defense (`jtiCache` + `source_boot_time` floor) lives in
`internal/federation/coordinator/blob.go` — inside an operator-only package the
donor must not import. `wire.Verify` deliberately does **not** enforce replay.
M4.1 extracts the donor-safe pieces:

- New `internal/federation/replay` (pure stdlib): the single-use `jti` TTL cache
  and the boot-time-floor helper, imported by **both** the coordinator source
  endpoint and the donor read-source endpoint. The coordinator's M4 `jtiCache` is
  refactored onto it (no behavior change).
- The coordinator-only **mint** stays in `internal/federation/tokens`; M4.1 adds a
  read-grant mint variant (donor-source/coordinator-dest direction) reusing
  `wire.SigningInput` + `AssembleToken`.

This both honors the donor dependency boundary (`scripts/check_node_deps.sh`) and
removes duplication before M5 generalizes donor↔donor repair.

### D-M4.1-5 — Donor egress (D11) takes its first real debit here, with conservative no-refund semantics.

M4's D-M4-6 said the D11 egress bucket's first debit lands in M5 because in M4 the
donor only *receives* bytes. M4.1 changes that: once a donor serves ciphertext to
the coordinator on a read-miss, the **donor sends bytes**, so the donor
read-source **must** debit the authoritative `bandwidth.Bucket` (D11). Precise
policy:

- The serve unit is the **complete envelope** (`blob_manifests.envelope_size`,
  D-M4.1-16) — a source either serves the whole envelope under a sufficiently
  large grant or serves nothing. So: **if `envelope_size > token.max_bytes`,
  refuse before serving** (no truncated serve); else `Bucket.Take(envelope_size,
  now)` **before** streaming. Never `min(envelope_size, max_bytes)` — that would
  under-account or imply a truncated serve is valid.
- Insufficient budget ⇒ refuse. Because this is a `GET /fed/v1/blob/{cid}` refusal
  (not a donor `/fail` POST), the endpoint returns a
  `wire.ErrorResponse{code:"budget_exceeded"}` using the same normalized string as
  the `wire.Fail` reason domain; the coordinator falls back to the next sourceable
  holder.
- **No refund in M4.1** if the stream later aborts. This is conservative and
  simple: it slightly underutilizes budget on client disconnects but denies an
  adversary an abort-to-evade-accounting vector. A reserve/refund model can come
  later if measurement justifies it.

The donor's **inbound** replication-receive traffic remains a separately named
budget (not folded into the D11 egress bucket), exactly as M4 stated.

### D-M4.1-6 — Coordinator donor-fetch is a backend *tier* behind `OpenBytes`; verify before decrypt.

The single read seam stays `storage.Service.OpenBytes`
(`pkg/coordinator/storage/blob.go`), consumed by `internal/api/handlers/blob.go`
via the stable `Reader` interface. Donor fetch is a new **tier**, not a second
public origin:

```
OpenBytes(cid):
  1. local bounded cache / Backend.Get      → hit ⇒ decrypt + serve (+ touch LRU)
  2. miss & storage mode permits:
       select sourceable holder             (D-M4.1-7, D-M4.1-8)
       SourceFetcher.Fetch(donor, cid, grant, maxBytes)   → bounded ciphertext
       VERIFY: deterministic re-import (importspec) → root-CID == cid
       decrypt via existing v1 envelope path
       serve
       re-cache per storage mode            (D-M4.1-9)
```

Verification reuses the coordinator's embedded backend +
`internal/ipfs/importspec` (it has go-cid, unlike the donor). **Unverified donor
bytes are never decrypted or served.** The verification import itself pins the
bytes locally (the `Backend.AddDeterministic` contract); D-M4.1-9 specifies what
happens to that pin per mode.

### D-M4.1-7 — Count **sourceable** holders, not bare `acked` holders, for commit / prune / read.

`pin_assignments(state='acked')` is necessary but not sufficient evidence that a
donor can serve a read *now*. A historically-acked node may be long unreachable
(its `nodes.status` can lag until M5's liveness sweeper exists), may not advertise
the read-source capability, or may have no reachable address. M4.1 defines an
interim **sourceable holder** predicate and uses it everywhere durability is
asserted:

```sql
sourceable_holder(cid) :=
  pin_assignments.state = 'acked'
  AND nodes.status IN ('active','suspect')
  AND nodes.trust_state <> 'suspended'
  AND nodes.last_seen_at > now() - read_source_stale_after
  AND nodes.advertised_capabilities @> ARRAY['read-source/v1']
  AND nodes.source_nebula_addr IS NOT NULL AND nodes.source_nebula_addr <> ''
```

`trust_state` (`probationary`/`trusted`/`suspended`, migration 0011) excludes a
suspended donor from commit quorum, prune safety floor, and read selection;
`probationary` still counts in M4.1 (a stricter important-class rule is deferred).

`advertised_capabilities` already exists (migration 0011) and is persisted at
register; M4.1 adds `read-source/v1` to the donor's advertised set and
`source_nebula_addr` (D-M4.1-10). Historically-acked holders remain *evidence*,
but **commit quorum and `prune_safety_floor` count sourceable live holders** so
the coordinator never commits/prunes against replicas that cannot actually serve.
`read_source_stale_after` is the interim freshness gate until M5 owns the full
5-state liveness sweeper.

### D-M4.1-8 — Read-source selection is best-link over the sourceable set.

The selector queries sourceable holders of the CID ⋈ `nodes`, ordered by
`reputation_score` desc (uses the existing
`nodes_reputation_idx WHERE status IN ('active','suspect')`), with rotation among
equal-reputation peers to spread load. On a fetch error or refusal it advances to
the next holder, bounded by `read_source_max_fallbacks_per_request` (D-M4.1-13).
Explicit VPS/residential **link-class** ordering is deferred (no column today) —
M4.1 ranks by reputation + liveness only; link-class is M5/M6.

### D-M4.1-9 — `coordinator_storage_mode`, with explicit per-mode pin/unpin semantics.

A new enum config knob, default-preserving:

- **`origin_copy`** (default): the coordinator keeps every blob's local copy and
  **never prunes**. Identical to today's behavior — no operator who does nothing
  sees a change.
- **`bounded_cache`**: the coordinator keeps a hot cache ≤
  `bounded_cache_max_bytes`; origin copies are prunable once safe (D-M4.1-12);
  reads re-cache. The eviction policy is a **size-aware two-queue / segmented-LRU
  (SLRU/2Q)**, *not* plain LRU: a newly donor-fetched object enters the
  **probationary** segment; an object hit again is promoted to **protected**;
  eviction **drains probationary (oldest `last_accessed_at`) first**, and only
  demotes/evicts the oldest **protected** entries if the protected segment exceeds
  its byte budget (`bounded_cache_protected_ratio`, default 0.80). This protects
  established hot objects from one-off scans / bulk cold-read waves / crawler
  traffic — the classic scan-pollution failure mode for Nova's large immutable
  blobs — while staying simple and DB-backed. (Plain LRU is acceptable only as an
  early-development fallback, never the normative policy; W-TinyLFU/ARC/GreedyDual
  are explicitly **not** adopted in M4.1.) `local_bytes` accounting + admission +
  eviction + the optional `bounded_cache_max_object_bytes` ceiling use
  `envelope_size` (D-M4.1-16). To avoid write-amplification, `last_accessed_at`
  (and promotion) is touched **at most once per CID per `lru_touch_interval`** —
  hot reads must not become DB churn.
- **`transient`**: the coordinator holds nothing beyond upload staging; every
  committed read is donor-backed.

Because verification imports pin (D-M4.1-6), the post-verify pin disposition is
mode-explicit:

```
origin_copy   : keep the verification pin as the local copy
bounded_cache : keep iff the cache admits it (enters probationary; SLRU/2Q above);
                refuse if envelope_size > bounded_cache_max_object_bytes; else Backend.Unpin
transient     : Backend.Unpin immediately after decrypt (unless an explicit
                emergency-retain policy fires, e.g. holder count below floor)
```

`local_present` in the projection means **Nova intends to retain a local pin** —
not raw block existence. `Backend.Unpin` removes the pin; Kubo GC reclaims blocks
later. **The projection, never raw block presence, drives storage policy.**

**Unpin timing (transient / direct-stream safety).** In `transient` mode the unpin
must not race the returned reader. For encrypted v1, `OpenBytes` reads + decrypts
into memory before returning, so unpin-after-decrypt is safe. For
`public_archival` unencrypted blobs `OpenBytes` may stream bytes directly, so the
implementation **wraps the returned reader and unpins on `Close`** (after the
reader is fully materialized/closed) — never before the serve completes.

### D-M4.1-10 — Durable storage-state projection + `durability_class`; authority discipline.

Crash-safe pruning and visibility gating need durable state that Kubo + memory
cannot provide. M4.1 lifts `migrations-frozen` for a single migration (`0013`)
adding:

```sql
CREATE TYPE blob_commit_state    AS ENUM ('staging','committed','failed');
CREATE TYPE coordinator_local_role AS ENUM ('origin','staging','cache','absent');
CREATE TYPE cache_segment        AS ENUM ('probationary','protected');  -- SLRU/2Q (D-M4.1-9)

CREATE TABLE blob_storage_state (
  cid               text PRIMARY KEY REFERENCES blobs(cid) ON DELETE CASCADE,
  commit_state      blob_commit_state      NOT NULL DEFAULT 'committed',
  durability_class  text                   NOT NULL DEFAULT 'normal'
                      CHECK (durability_class IN ('important','normal','cache')),
  local_role        coordinator_local_role NOT NULL DEFAULT 'origin',
  local_present     boolean                NOT NULL DEFAULT true,
  local_bytes       bigint                 NOT NULL DEFAULT 0,
  cache_segment     cache_segment,         -- NULL unless local_role='cache' (SLRU/2Q segment)
  committed_at      timestamptz,
  last_accessed_at  timestamptz,
  prune_eligible_at timestamptz,
  last_refetch_at   timestamptz,
  updated_at        timestamptz            NOT NULL DEFAULT now()
);

ALTER TABLE nodes ADD COLUMN source_nebula_addr text;  -- donor read-source reachability
```

- **Authority discipline.** `pin_assignments(state='acked')` + node
  freshness/capability/address (the sourceable predicate, D-M4.1-7) are the
  **authoritative** holder source of truth. Any holder *count* materialized in a
  projection is a **rebuildable cache**, never authority.
- **`durability_class`** is stored (not re-derived from product/parent forever)
  because M5 placement/target counts need the same concept. Defaults on commit:
  `parent_cid IS NULL ⇒ important`; derivative ⇒ `normal`; explicitly
  transient/generated ⇒ `cache`. The class selects the replication target (the
  existing `Replication.Factor.{Important,Normal,Cache}` = 5/3/2, used by M5
  placement) **and** the class's `commit_quorum` entry — a separate, smaller value
  (the number of *sourceable* acks required to commit, bounded above by the class
  factor; conservative default e.g. 2).
- **Backfill.** The migration seeds existing blobs as
  `committed / origin / local_present=true` with the derived class, so current
  deployments are unchanged.

### D-M4.1-11 — Async staging commit gate (`require_replication_quorum_before_commit`), default-off, with explicit read/API semantics.

Donor acks are poll-driven (seconds–minutes); blocking the upload HTTP response on
quorum would couple user-facing latency to donor poll/import time and tie up
assembly slots. So the gate is **asynchronous**:

```
upload accepted → local staging import → blob row + storage_state(commit_state='staging')
  → minimal admission assigner creates donor assignments (D-M4.1-17)
  → replication proceeds (M3/M4)
  → durability reconciler observes sourceable acked quorum (per durability_class)
  → commit_state='committed', committed_at set; product OnCommitted hooks fire NOW
  → reads become visible; staging becomes prune-eligible (D-M4.1-12)
  → permanent inability to reach quorum ⇒ commit_state='failed' (operator-surfaced)
```

- **Assignment source.** A fresh staging upload only progresses if *something*
  creates donor assignments — and today `AssignPin` is a manual `novactl`/test
  seam (the M5 scheduler does not yet exist). M4.1 therefore includes a **minimal
  admission assigner** (D-M4.1-17); the design must not assume M5's scheduler.
- **Hook lifecycle.** The current `Put` fires `s.hook.OnCommitted` immediately
  after the DB transaction (`put.go:164`). Under the gate that transaction only
  writes a `staging` row, so OnCommitted hooks (e.g. derivative prewarm) **must
  not fire at staging-write time** — they fire only when the durability reconciler
  transitions `commit_state` staging→committed. Staging-time product work, if any,
  uses a separate internal hook whose outputs are **not** public-read-visible
  (D-M4.1-14).

- **Default-off.** When `require_replication_quorum_before_commit` is false (the
  default, matching `origin_copy`), uploads commit immediately as today. The gate
  is the required posture for serious `bounded_cache`/`transient` deployments.
- **Read/API semantics under the gate.** Public/anonymous/signed-URL reads of a
  `staging` blob return **404** (no existence leak; preserves the current privacy
  posture). Owner/admin metadata APIs may show `staging`. The upload response
  returns the CID plus a `durability_state` field; when the gate is on the
  multipart create path returns **202 Accepted** (instead of 201) with
  `durability_state:"staging"`, so clients can distinguish accepted-but-not-yet-
  durable from committed. The reconciler runs on the existing orchestrator tick
  cadence (`commit_reconciler_interval`); it is idempotent and crash-safe via the
  projection.

### D-M4.1-12 — Origin/staging pruning is safety-floor-gated and never treats cache as a replica.

A pruner reclaims local copies once donors are the durable substrate:

- **Prune condition.** For a `committed` blob in mode `{bounded_cache,transient}`
  with **`sourceable_holder_count(cid) >= prune_safety_floor`** *at the moment of
  pruning*: `Backend.Unpin` the local copy → Kubo GC reclaims; update the
  projection (`local_present=false`, `local_role` per mode, record prune).
  `bounded_cache` additionally evicts past `bounded_cache_max_bytes` using the
  **SLRU/2Q** order (D-M4.1-9) — drain probationary oldest-`last_accessed_at`
  first, demote/evict protected only if it exceeds `bounded_cache_protected_ratio`
  — and only for blobs whose sourceable holder count is safe.
- **Never below the floor.** The floor is re-evaluated every pass against the
  *live sourceable* count. If a local copy still exists and the count would drop
  below the floor, the pruner retains/re-pins.
- **Post-prune holder loss (honest limits).** If the local copy is **already
  absent** and live sourceable holders fall below the floor, M4.1 does **not**
  count the coordinator cache/origin as a replica and does **not** pretend the
  object is healthy. It **alerts**, **stops further pruning** for that CID/class,
  and may attempt an **emergency donor re-fetch into local cache**. Durable
  re-replication (restoring holders) is **M5 healing's** job. The coordinator
  cache/origin copy **never counts toward `R`** (consistent with the master
  design).

### D-M4.1-13 — Donor-backed reads are protected integration points.

A cache miss now crosses a remote donor integration point before serving a user
response — the classic surface for cascading failure, slow responses, blocked
goroutines, and dogpiles. `OpenBytes` donor fetch is wrapped in production
failure-containment, configured by safe defaults (advanced knobs, D-M4.1-15):

```
read_source_timeout                    per-fetch deadline
read_source_max_fallbacks_per_request  bounded holder fallback (no unbounded retry in OpenBytes)
read_source_bulkhead                   coordinator-wide max concurrent donor fetches
per_donor_fetch_limit                  max concurrent fetches per donor
donor_circuit_breaker                  per-donor breaker; open ⇒ skip to next holder / fail fast
cache_fill_singleflight                per-CID single-flight: N concurrent misses ⇒ 1 fetch + N-1 waiters
fail_fast_when_no_sourceable_holder    no holder ⇒ fail fast (status per below), no thrash
```

**Unavailable-read status semantics** (don't lie about existence):

```
staging public read                              → 404 (no existence leak)
known, committed, authorized blob —
  no local copy AND no sourceable holder         → 503 Service Unavailable
private unauthorized read                        → existing auth behavior
```

A committed object that exists but is momentarily unsourceable is **503**, never
404 — the latter would falsely tell a client an existing object is gone.

Single-flight per CID is the direct defense against a cold-read dogpile (100
simultaneous misses for one CID trigger one donor fetch, not 100). Donor egress
budget (D-M4.1-5), coordinator fetch concurrency, and cache-fill single-flight are
treated as **flow-control mechanisms**, not incidental implementation details.

### D-M4.1-14 — Transform re-fetch is transparent; no staging backdoor.

Derivative generation reads the parent through `OpenBytes`
(`pkg/coordinator/storage/derivative.go`, `internal/jobs/kinds/derivative_prewarm.go`),
so a pruned parent **re-fetches from a donor transparently** through the same tier
(D-M4.1-6) and re-caches per mode. **Guard:** a derivative of a `staging`
(not-yet-committed, not-visible) parent must **not** be served via public
transform routes — transform generation either waits for the parent's commit or
uses an explicit staging-authorized internal path. Public transform routes must
never become a backdoor that exposes a not-yet-committed original. Concretely:
under the gate, the product `OnCommitted` hooks that drive prewarm fire **only on
the durability transition to `committed`** (D-M4.1-11), not on the staging write —
so derivative generation cannot run against, or publish from, a `staging` parent.

### D-M4.1-15 — Observability is evidence-shaped events **plus** a USE resource model.

Per Nova's slog convention (Prometheus promotion is M7), M4.1 emits evidence
events *and* resource gauges/counters that map cleanly to Prometheus later:

| Seam | slog events (fields) |
|---|---|
| coordinator read | `storage.read.cache_hit`/`cache_miss` (`cid`); `storage.read.donor_fetch` (`cid`,`holder`,`bytes`,`dur_ms`); `storage.read.donor_fetch_failed` (`reason`,`fallback`) |
| commit gate | `storage.commit.staged`/`committed`/`failed` (`cid`,`durability_class`,`quorum_latency_ms`) |
| pruning | `storage.prune.applied` (`cid`,`holders`,`bytes_reclaimed`); `storage.prune.skipped_floor` (`cid`,`holders`) |
| donor source | `node.source.served` (`cid`,`bytes`,`dur_ms`,`dest`); `node.source.refused` (`reason` incl. `budget_exceeded`) |

**USE resources** (utilization / saturation / errors) to track: coordinator
donor-fetch worker pool; per-donor fetch concurrency; cache bytes + item count +
admission/eviction; Kubo Add/Get/Unpin latency+errors; DB holder-selection
latency+errors; commit-reconciler + pruner queue depth; donor egress bucket tokens
+ refusals; donor source-server concurrency.

### D-M4.1-16 — Transfer, budget, and cache accounting use **envelope size**, not `blobs.byte_size`.

Nova stores two sizes: `blobs.byte_size` is the **product/plaintext** size
(written as `len(buf)`, `put.go:204`), while `blob_manifests.envelope_size` is the
byte length actually **imported, transferred, pinned, cached, and budgeted**
(`add.EnvelopeSize`, `put.go:216`). Today the federation transfer path reasons
about the wrong one: `GetBlobSize` and the source preflight's `GetBlobByteSize`
both `SELECT byte_size FROM blobs` — plaintext — even though M4 serves the
ciphertext envelope (so for encrypted blobs the `max_bytes` preflight under-counts
by the AEAD/framing overhead). **M4.1 corrects this latent M4 defect:** all
federation read grants, source preflight checks, donor egress debits
(D-M4.1-5), `max_transfer_bytes` limits, `blob_storage_state.local_bytes`, and
bounded-cache admission/eviction use **`blob_manifests.envelope_size`** (a new
`GetEnvelopeSize` query). A source refuses before serving if
`envelope_size > token.max_bytes`; it never serves a truncated envelope and never
debits `min(envelope_size, max_bytes)`. (`pin_changes.byte_size` is advisory M3
sizing and may be reconciled to envelope size opportunistically; it is not on the
transfer-safety path.)

### D-M4.1-17 — A minimal admission assigner makes the commit gate operative before M5.

`AssignPin` is today a manual `novactl`/test seam; the M5 healing/placement
scheduler does **not** exist yet, so a fresh staging upload would never replicate
and the commit gate would be architecturally present but **operationally inert**.
M4.1 adds a deliberately **minimal admission assigner** that, for a new staging
blob, selects `R(durability_class)` source-capable donors by reputation/liveness
(the sourceable predicate, sans the CID-acked clause) and calls the **existing
`AssignPin` seam**. It has **no** failure-domain anti-affinity, **no** healing,
**no** repair — those are M5, which **replaces/generalizes** this assigner via the
same seam. When the gate is off, the assigner is best-effort (uploads still commit
immediately); when on, it is the engine that drives a blob toward quorum.

### D-M4.1-18 — Donor read-source requires a current coordinator repair public key; rotation is fail-closed.

The donor verifies read grants with the coordinator's repair public key learned
via M4 heartbeat/registration. The read-source server and the outbound agent are
separate goroutines/boundaries, so the key must be made available across them
(persisted in donor state or shared in-process). Until a current key is known the
source endpoint refuses with `source_unavailable` (it never serves unverified).
On **key rotation** the server accepts old+new within the same TTL/grace window
the repair grants use, or **fail-closes until refreshed** — it must not silently
accept a stale key or reject all grants without signalling.

### D-M4.1-19 — Reject unsafe config combinations at load.

The loader validates the new knobs (shallow checks, consistent with existing
loaders) so an operator cannot configure data loss:

```
transient                  ⇒ require_replication_quorum_before_commit = true
bounded_cache & gate=false ⇒ allowed only with a warning; refused if pruning is enabled
1 <= commit_quorum[class]  <= replication.factor[class]
prune_safety_floor         >= commit_quorum[class]   (for prunable classes)
prune_safety_floor         <= replication.factor[class]   (else pruning never occurs — warn)
read_source_stale_after    >= 2–3× heartbeat interval + clock-skew allowance
bounded_cache_max_bytes    >  max_transfer_bytes (or the largest admitted blob), unless
                              large blobs are intentionally non-cacheable
0 < bounded_cache_protected_ratio < 1                  (SLRU/2Q segment split)
bounded_cache_max_object_bytes <= bounded_cache_max_bytes   (when set, > 0)
```

These governors keep the operator surface simple while preventing the unsafe
combinations that would let pruning outrun durability.

## Wire reconciliation

- **New capability constant:** `CapReadSource = "read-source/v1"` (donor →
  coordinator ciphertext serve). Distinct from the reserved
  `CapRepairStream = "repair-stream/v1"` (M5 donor↔donor). The donor advertises
  `CapReadSource`; the coordinator records it and uses it in the sourceable
  predicate.
- **Token direction (reused types).** `wire.Claims` is unchanged — M4.1 uses it
  with `source_node_id = donor`, `dest_node_id = reserved-coordinator-source-id`,
  and now **enforces** `assignment_id`/`generation` at the source (the donor).
  Read grants are minted by the coordinator with `wire.SigningInput` +
  `AssembleToken`.
- **`RegisterRequest`/`HeartbeatRequest`** gain `source_nebula_addr` (the donor's
  reachable read-source address); the coordinator persists it on `nodes`.
- **Upload response** gains `durability_state`; the gate-on multipart create path
  returns 202 (D-M4.1-11).
- **`wire.Fail`** reason domain is unchanged — `budget_exceeded`,
  `blob_unavailable`, `source_unauthorized`, `cid_mismatch`, `network_error`,
  `kubo_error` already cover the new refusals.
- **Endpoint-only `ErrorResponse` codes.** `GET /fed/v1/blob/{cid}` is a synchronous
  fetch, not a donor `/fail` POST, so it returns `ErrorResponse{code}` — reusing
  the `wire.Fail` strings where they apply (`source_unauthorized`,
  `blob_unavailable`, `budget_exceeded`) **plus** two endpoint-only codes,
  `source_unavailable` (no current coordinator pubkey, D-M4.1-18) and
  `blob_too_large` (`envelope_size > max_bytes`, D-M4.1-16). These two are **not**
  donor `/fail` reasons.

The `wire` package stays dependency-free (donor-safe).

## Architectural notes

### Donor read-source server (net-new, coordinator-only)

`internal/node/source` registers `GET /fed/v1/blob/{cid}` on a Nebula-bound mTLS
listener stood up by `cmd/node`. It imports only donor-boundary-safe packages —
`wire`, `transport`, `replay`, `ipfsclient`, `bandwidth` — never `internal/ipfs`
or any operator-only code. It authenticates the coordinator **client** role,
runs the D-M4.1-3 verification chain (including `assignment_id`/`generation`
against local M4 progress state and `ipfsclient.Has`), debits the D11 egress
bucket (D-M4.1-5), then streams the pinned envelope under an `io.LimitReader`.

### Coordinator read tier + selection

`OpenBytes` gains the miss tier (D-M4.1-6). Holder selection is a new sqlc query
over the sourceable predicate (D-M4.1-7/-8). Fetch reuses the `SourceFetcher`
seam (M4 D-M4-12), now targeting a donor `source_nebula_addr` with a
coordinator-minted read grant, over the coordinator's federation **client**
identity. Verification reuses the embedded backend + `importspec`. All fetches
run inside the D-M4.1-13 containment (timeout, bulkhead, per-donor breaker,
per-CID single-flight, bounded fallback).

### Durable state + reconciler + pruner

`blob_storage_state` (D-M4.1-10) is written on commit, touched on read (LRU),
flipped by the **durability reconciler** (D-M4.1-11), and mutated by the
**pruner** (D-M4.1-12) — both background loops on the orchestrator tick cadence,
idempotent and crash-safe. Holder counts they consume are computed live from the
sourceable predicate (authority discipline).

### Config (coordinator + donor)

Coordinator (`internal/config` + hot-reload `reload.Store` +
`config_admin.fieldEffect` + `/settings`): `coordinator_storage_mode`,
`bounded_cache_max_bytes`, `prune_safety_floor`,
`require_replication_quorum_before_commit`, `commit_quorum` (per durability
class), `read_source_stale_after`, the SLRU/2Q tuning
(`bounded_cache_protected_ratio` default 0.80, optional
`bounded_cache_max_object_bytes`), and the D-M4.1-13 containment knobs. Donor
(`node.yaml`): the read-source listen address + `source_nebula_addr`
advertisement, and the egress budget already present from M4.

**Simplicity of the operator surface (normative).** Exactly **four** storage
knobs are first-class in `/settings`: `coordinator_storage_mode`,
`bounded_cache_max_bytes`, `require_replication_quorum_before_commit`,
`prune_safety_floor`. All cache-tuning / timeout / bulkhead / circuit-breaker /
interval knobs (`bounded_cache_protected_ratio`, `bounded_cache_max_object_bytes`,
`read_source_timeout`, `read_source_max_fallbacks`, `read_source_stale_after`,
`read_source_bulkhead`, `per_donor_fetch_limit`, `lru_touch_interval`,
`commit_reconciler_interval`, `pruner_interval`) remain **advanced/hidden with
safe defaults** — an operator must not have to tune a dozen distributed-systems
controls to avoid data loss. The D-M4.1-19 validator backstops unsafe
combinations regardless of which knobs are touched.

## Trust-model notes (Phase 2 adversaries; canonical text in `THREAT_MODEL.md`)

- **Donor-blind preserved (T1.26).** Donors hold and serve only ciphertext, and
  only to the authenticated coordinator; the coordinator decrypts and serves every
  user byte. M4.1 does not add a donor plaintext path or a client-direct donor
  read path.
- **Read-grant forgery/replay (D1).** Read grants are coordinator-minted and
  donor-verified (donors hold only the public key, Ed25519, asymmetric);
  source/dest/cid/assignment/generation/max_bytes-bound; single-use `jti` +
  `source_boot_time` floor + short TTL via the shared `replay` helper.
- **Wrong/tampered donor bytes.** Caught by deterministic re-import + root-CID
  compare **before** decrypt (D-M4.1-6); unverified bytes are never served.
- **Bandwidth exhaustion (D11).** The donor's authoritative egress bucket debits
  per serve and refuses `budget_exceeded` (D-M4.1-5); coordinator fetch
  concurrency, per-donor breaker, and per-CID single-flight bound coordinator-side
  amplification (D-M4.1-13).
- **False durability / premature pruning.** Commit and prune count **sourceable
  live** holders — acked **and** non-suspended **and** fresh **and**
  read-source-capable **and** addressable (D-M4.1-7) — not historical acks; the
  safety floor is re-evaluated each pass and the cache never counts as a replica
  (D-M4.1-12), so pruning never reduces real durability below the floor.
- **Size-accounting confusion.** Transfer grants, preflight, egress debit, and
  cache accounting use the **envelope** size, and a source serves the whole
  envelope or nothing (D-M4.1-16) — no truncated serve, no plaintext/ciphertext
  size mismatch.
- **Identity confusion.** The coordinator client role is a distinct cert identity,
  not an overloaded source constant or a fake `nodes` row (D-M4.1-2); donor
  read-source accepts only the coordinator role.

## Exit criterion

With `coordinator_storage_mode=bounded_cache` and
`require_replication_quorum_before_commit=true`: a fresh upload is accepted
(`202`, `durability_state:"staging"`) and is **not** publicly readable (404) while
`staging`; the **minimal admission assigner** creates donor assignments; donors
replicate and ack to the configured per-class quorum of **sourceable** holders;
the durability reconciler flips it to `committed`, **then** the product
`OnCommitted` prewarm fires, and public reads succeed. The pruner then unpins the
origin (sourceable holders ≥ `prune_safety_floor`); `Backend.Has` is false
locally; a fresh read **selects a sourceable holder, fetches the full ciphertext
envelope over a coordinator-minted read grant (debiting the donor's egress
budget), verifies by deterministic re-import before decrypt, decrypts, serves, and
re-caches**, and a second read is a local cache hit. A transform of the pruned
parent re-fetches transparently and is never served from a `staging` parent.
Dropping sourceable holders below the floor halts further pruning and alerts; a
committed blob with no sourceable holder returns **503**, never 404; the cache is
never counted as a replica. `donor-deps-boundary` stays green (the read-source
server adds only the reviewed allowlist entries); `migrations-frozen` lifts only
for `0013`. **No donor↔donor repair, healing scheduler, or client-direct donor
reads are in this milestone — those are P2-M5.**
