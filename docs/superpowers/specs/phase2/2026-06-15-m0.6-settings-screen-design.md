# P2-M0.6 — Admin Settings screen (`web/admin`)

## Context

M0.4 built the runtime-config **backend** explicitly as *"the backend the future
setup wizard (M0.5) and admin settings modal (M0.6) call"*: an operator-only API
(`GET`/`PATCH`/`PUT /api/v1/admin/config`) that reads the effective config with
per-field effect/source metadata and writes validated changes to the canonical
`operator.yaml`, hot-reloading the fields that can apply live. M0.5 then shipped
the first-run wizard UX (consequence copy, jargon info-buttons, tri-state paranoid
delineation) on top of the M0.2 privacy-posture model. M0.6 is the **last
P2-M0.x remediation item**: the admin SPA UI that finally drives the M0.4 API at
runtime, so an operator can see and tune the config of a *running* node without
hand-editing YAML or shelling into `novactl`.

**No backend work.** M0.4 delivered the full read/update/validate/hot-reload/audit
contract and `novactl config` parity. M0.6 is pure `web/admin` frontend that
consumes that contract as-is.

**Scope.** A new operator-only Settings **screen** with curated, fully-explained
controls for the settings an operator actually tunes at runtime — the privacy
constituents, the live upload limits, CORS, and public-uploads/ToS — plus a
read-only viewer of the *entire* effective config with effect/source badges.
**Non-goals:** no generic editable form over every `operator.yaml` field (the long
tail — `auth`/`tls`/`moderation` and the `env-only-inert` sections — stays
`novactl config` / hand-edit); no new endpoint, secret, or trust boundary; no
extraction of a shared component package (the M0.5 privacy components are ported
into `web/admin`, see "Component reuse"). Depends on M0.4 + M0.5.

## Decisions (ratified with Bug, 2026-06-15)

- **Screen, not a literal modal.** The roadmap nicknames M0.6 "Admin Settings
  modal," but the admin SPA's convention is route-per-surface (Blobs, Moderation,
  Keys…) with Radix modals reserved for discrete actions. The privacy tree + four
  sections + an effective-config viewer + save/validation flow want room to
  breathe, so M0.6 ships as an operator-only `/settings` **route + nav entry**
  (gated exactly like Keys: `roles: ['operator']` in `Shell.tsx` NAV +
  `isOperator` in-screen). "Modal" is read as shorthand for the config UI.
- **Curated editing + read-only full visibility.** Hand-crafted, explained
  controls only for the high-impact / routinely-tuned settings; a collapsible
  read-only "Effective config" table renders every leaf for transparency. No
  generic editable form — it would expose `restart`/`env-only-inert` knobs without
  the consequence copy that is the entire point of the P2-M0.x track, and invite
  edits to fields no runtime consumer reads yet.
- **`novactl` stays the full-power path.** The screen makes the common and
  dangerous decisions legible; `novactl config get/set/apply` remains the way to
  edit the long tail. The viewer says so explicitly.
- **The effect split is the spine of the UX.** Every curated control wears a
  **live** (applies immediately) or **restart-required** badge. The badge renders
  from the GET `fields[path].effect` metadata (the live source of truth, including
  env-shadowing); the registry's static `effect` is only a fallback/test
  expectation, and a test asserts the two agree so they can't silently drift.
- **Warn-free saves, with the runtime webhook caveat.** Unlike the M0.5 first-run
  wizard (where webhooks never exist), a *running* node can have outbound webhooks
  configured — and `ApplyPrivacyPreset` warns whenever `paranoid:true` with a
  non-empty `webhooks` list (verified in `internal/config/paranoid.go`). So the
  screen derives `auth.paranoid = (record-off ∧ retention≤1 ∧ dht-private ∧
  webhooks-empty)` and renders the parent **indeterminate ("partial")** when
  webhooks are configured, with copy that the full preset can't be represented
  warning-free from this screen (clear `webhooks` via `novactl`/hand-edit). With
  that stricter derivation, **no save the screen makes can trip an
  `ApplyPrivacyPreset` startup WARN**. The screen *surfaces* the GET
  `privacy_warnings`; saving resolves drift from the **editable** constituents, but
  a webhook-induced warning persists until webhooks are cleared.

## The M0.4 contract M0.6 consumes (no change)

`GET /api/v1/admin/config` →
```jsonc
{
  "version": 7,                       // etag source for optimistic concurrency
  "config":  { /* full operator.yaml as a nested map */ },
  "privacy_warnings": ["..."],        // from ApplyPrivacyPreset
  "fields":  { "<dotted.path>": { "effect": "live|restart|env-only-inert",
                                  "source": "yaml|env", "shadowed_by_env": true } }
}
```
`PATCH /api/v1/admin/config` — JSON Merge Patch (RFC 7396) body; optional
`If-Match: <version>` → **409** `config_conflict` when stale; **422**
`config_invalid` (message) on validation failure, **persists nothing**; **200**
returns the same shape **plus** `restart_required: [<dotted.path>…]`. (`PUT` is the
full-replace power path — M0.6 uses **PATCH only**.)

The merge is RFC-7396, so a JSON `null` **deletes** a key. The patch builders
therefore only ever emit concrete values for the fields they own; the minimal-diff
helper never emits `null` (see Data flow).

## The effect split (curated fields, from the M0.4 table)

| Section | Fields | Effect |
|---|---|---|
| Upload limits | `uploads.max_upload_size_bytes`, `uploads.max_concurrent_assembly`, `uploads.limits.max_concurrent_global`, `uploads.limits.max_concurrent_per_session`, `uploads.limits.max_files_per_session` | **live** |
| Cross-origin uploads | `uploads.cors.enabled`, `uploads.cors.allowed_origins` | **live** |
| Privacy & hardening | `coordinator.record_source_ip`, `source_ip_retention_days`, `coordinator.public_ipfs_dht`, `auth.paranoid` (+ read-only `webhooks`, metrics) | **restart** |
| Public uploads / ToS | `uploads.public_uploads`, `tos_url` | **restart** |

So the screen tells one clean story: **Upload limits + CORS apply immediately;
Privacy and Public-uploads changes need a coordinator restart.** The
`restart_required` array returned on save drives a banner confirming exactly which
changed leaves are waiting on a restart.

## Sections

### 1. Privacy & hardening (restart)
A port of M0.5's `ParanoidSection` tri-state model: a parent "Harden privacy
(paranoid)" select-all over three child toggles plus two read-only info rows.

- **Children (protective = checked):** don't-record-uploader-IPs
  (`coordinator.record_source_ip = !checked`), keep-IP-logs-short
  (`source_ip_retention_days`, see below), keep-CIDs-off-public-DHT
  (`coordinator.public_ipfs_dht = !checked`). Same polarity inversions as the M0.5
  wire-mapping table.
- **Retention preserves the operator's raw value.** Draft state holds
  `retentionDays: number`; the toggle's protective test is `retentionDays ≤ 1`.
  Checking sets `retentionDays = 1`; unchecking restores the loaded value when it
  was `> 1`, else defaults to `30`. This stops the first-run binary `checked?1:30`
  from silently clobbering a hand-edited value like `7` (which loads as
  not-protective): the minimal patch emits `source_ip_retention_days` **only when
  the number actually changed**.
- **Parent tri-state (webhook-aware).** Protective ⇔ all three children protective
  **and** `webhooks` empty. all-protective → checked; none → unchecked; mixed or
  webhooks-present → indeterminate. Clicking the parent toggles the three children;
  it cannot clear webhooks (read-only here), so when webhooks exist the parent stays
  indeterminate with copy: *"Outbound webhooks are configured, so the full paranoid
  preset can't be represented warning-free here — clear `webhooks` via `novactl
  config`."* On save, `auth.paranoid = (record-off ∧ retention≤1 ∧ dht-private ∧
  webhooks-empty)`, so the persisted file is WYSIWYG and **no save the screen makes
  trips a startup privacy WARN** (verified against each `ApplyPrivacyPreset` warn
  branch in `paranoid.go`, including the webhook branch the M0.5 first-run case
  never hit).
- **Info rows (read-only):** webhooks (rendered from GET `config.webhooks` —
  count/presence; editing destinations is out of scope, `novactl`/hand-edit) and
  metrics-loopback-only (no `operator.yaml` field; startup-enforced).
- **Drift banner:** GET `privacy_warnings` render as a warning `Banner` atop the
  section. Saving re-derives the editable constituents WYSIWYG, so it **resolves
  drift from those** — but a warning caused by configured `webhooks` persists until
  webhooks are cleared (the parent's indeterminate state + copy make this explicit).

### 2. Cross-origin uploads (live)
`uploads.cors.enabled` toggle + an `uploads.cors.allowed_origins` add/remove list
(text input + chips with an `×`). Consequence copy reuses the M0.3 framing:
**CORS constrains browsers, not `curl`** — it is not an access-control boundary, it
only governs which web origins a browser will let script the upload endpoint.

**Origins are exact-matched by the middleware** (`cors.go` stores `allowed_origins`
verbatim and does a map lookup of the browser's `Origin` header — no normalization),
so the input normalizes each entry to a bare origin via `new URL(input).origin`
(scheme + host + optional port), previews the normalized value before adding, and
**rejects** path/query/trailing-slash or non-`http(s)` input — a `https://example.com/`
entry would never match a real browser origin. When `enabled` is true with an
**empty** origin list (matches nothing — enabled-but-useless), the section warns and
disables Save. The other CORS fields (`allowed_methods`/`allowed_headers`/
`exposed_headers`/`allow_credentials`) are left to the backend's tus defaults and
the effective-config viewer; not surfaced as editable controls.

### 3. Upload limits (live)
Number inputs for `uploads.max_upload_size_bytes` (with a human-readable MiB hint
beside the raw byte field), `uploads.max_concurrent_assembly`, and the three
`uploads.limits.*` caps. The consequence note is **"leave positive — zero or
negative is normalized by the config loader to Nova's default"** (verified:
`applyUploadDefaults` fills every `≤0` upload field with its `Default*` constant
*before* persist, so a `0` never reaches the live snapshot as "unbounded" for these
fields — it becomes the default). Client-side validation is advisory; the backend's
load-path floors are authoritative (422 surfaced inline).

### 4. Public uploads / ToS (restart)
`uploads.public_uploads` toggle with a strong abuse-surface `ConsequenceNote`,
coupled with a `tos_url` text input. **T1.20 is a backend hard-fail**: enabling
public uploads without `tos_url` returns 422 `config_invalid`. The UI **disables
Save** with an inline message for the obvious local case
(`public_uploads && tos_url.trim() === ''`); the 422 handler stays as the backstop
for stale-load / race cases and as the authoritative gate.

### 5. Effective config (read-only, full surface)
A collapsible table rendering **every** leaf path from the GET `config` map with:
its value, an `effect` badge (`live`/`restart`/`env-only-inert`), a `source` badge
(`yaml`/`env`), and an env-shadow marker — driven entirely by the GET `fields`
metadata. Includes a note: *"This screen edits high-impact settings. The full
surface is editable via `novactl config set/apply`."* This is what keeps curated
editing from being a dead end: full transparency without a fragile generic editor,
and it directly answers the operator questions M0.4's metadata was built for
("yaml or env?", "why didn't my change take effect?", "what needs a restart?").

## Data flow & save semantics

- **Non-operator guard:** the GET endpoint is operator-only, so for a non-operator
  (`user.role !== 'operator'`) the screen renders an operator-only notice and **does
  not fire `getConfig`** (no avoidable 403). The nav entry is already hidden; this
  covers direct navigation to `/admin/settings`.
- `useQuery(['config'])` → GET. Curated controls render from a **local draft**
  seeded off the loaded `config`; the read-only viewer renders straight from the
  query data.
- **Save** computes a **minimal JSON Merge Patch** (`mergePatch.ts`) from
  draft-vs-loaded — only changed leaves, nested under their section objects, never
  emitting `null`. The privacy section contributes its explicit child values +
  derived `paranoid`. `PATCH` is sent with **`If-Match: <loaded version>`**.
  - **200** → success toast; reseed the draft and the query cache from the returned
    `config` + `version`; if `restart_required` is non-empty, show a persistent
    "restart required for: …" banner listing the changed restart-class leaves.
  - **409 `config_conflict`** → banner "Configuration changed since you opened this
    screen" + a Reload action (re-GET, discard local draft).
  - **422 `config_invalid`** → inline error from the backend message; draft
    untouched (nothing persisted).
- **Discard** resets the draft to the loaded `config`.
- This matches the M0.4 guarantees rather than inventing a parallel frontend
  validation universe — the screen trusts the backend's single validation path.

## Component reuse

M0.5's `InfoTerm` / `ConsequenceNote` / `ParanoidSection` / `glossary.ts` live in
`web/setup` — a **separate hermetic Vite bundle**. There is no shared component
package today, and the two SPAs already duplicate primitives (`ui.tsx`, fonts,
CSS-module conventions). M0.6 therefore **ports adapted copies** into
`web/admin/src/settings/` rather than extracting a package (out of scope; would
touch the build graph and the hermetic-bundle gates for both SPAs). The ports are
small and self-contained: the glossary trims to the terms this screen uses
(paranoid, source-IP recording, IP retention, public IPFS DHT, CORS, public
uploads, ToS); `InfoTerm`/`ConsequenceNote` are copied near-verbatim;
`ParanoidSection` keeps its tri-state logic but takes draft values/handlers as
props instead of the wizard `FormState`.

A small **settings registry** (`registry.ts`) describes each curated field as
`{ path, label, description, consequence?, group, effect, read(config), patch(value) }`
— a data table the section components and the minimal-patch builder share. This is a
typed registry, **not** a generic form generator: the section components own their
layout and copy; the registry owns the path↔value plumbing.

## Frontend structure

New `web/admin/src/settings/`:
- `registry.ts` — curated field descriptors (path, effect, read/patch, copy).
- `glossary.ts`, `InfoTerm.tsx`, `ConsequenceNote.tsx` — ported from M0.5.
- `ParanoidSection.tsx` — ported tri-state parent + children + info rows.
- `CorsSection.tsx`, `UploadLimitsSection.tsx`, `PublicUploadsSection.tsx` — the
  curated sections.
- `EffectiveConfigViewer.tsx` — read-only full-surface table off `fields`.
- `mergePatch.ts` — minimal JSON Merge Patch from draft-vs-loaded (no `null`).
- `settings.module.css` — checkbox/toggle, child-tree connectors, badges, chips,
  origin list (the existing `ui.tsx` has no checkbox/toggle primitive; native
  inputs styled here).

`web/admin/src/screens/Settings.tsx` is the screen shell: the `useQuery`/draft
state, the save mutation, the section composition, and the save/discard footer. It
stays thin by delegating each section to its component.

## API + types changes

- `api/types.ts` — add `ConfigResponse` (`version: number`, `config:
  OperatorConfig`, `privacy_warnings: string[]`, `fields: Record<string,
  FieldMeta>`, `restart_required?: string[]`), `FieldMeta` (`effect`, `source`,
  `shadowed_by_env?`), and a structurally-typed `OperatorConfig` covering the
  curated paths (the long tail stays `Record<string, unknown>`-ish for the viewer).
- `api/client.ts` — add `getConfig()` → GET, and `patchConfig(patch, ifMatch?)` →
  PATCH carrying `If-Match`. The existing `send` helper builds only a content-type
  header, so `patchConfig` either extends `send` to accept extra headers or issues
  its own `fetcher` call through the same `parse<T>`. `ApiError.status`/`.code`
  already let the screen branch on 409/422.
- `app/App.tsx` — `<Route path="settings" element={<SettingsScreen />} />`.
- `app/Shell.tsx` — a NAV entry `{ to: '/settings', label: 'Settings', glyph: '⚙',
  roles: ['operator'] }`.

## Security / privacy notes

The screen is operator-only: the SPA gates the nav entry and screen via
`isOperator` (the `RequireAuth` wrapper handles the session), but enforcement is
the M0.4 API's `RequireRole("operator")` on the admin vhost — the API rejects a
non-operator regardless of the SPA gate. It reads and writes **no secrets** — the M0.4 surface
exposes only non-secret paths/values, and the effective-config viewer shows exactly
what the API returns. Every save is audit-logged by the M0.4 handler
(`config.updated` with an old→new diff). The privacy controls can only move
settings within the range M0.2 already sanctions, and saving them **narrows** the
warn surface (re-deriving `paranoid` WYSIWYG); the screen surfaces the same
consequence copy the coordinator would otherwise only log — strictly more informed
consent. Optimistic concurrency (`If-Match`) prevents a stale overwrite of an
out-of-band edit; the backend's validate-before-persist + atomic write mean a
rejected save leaves the running config intact.
