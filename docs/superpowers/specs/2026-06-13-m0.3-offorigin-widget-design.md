# P2-M0.3 ŌĆö Off-origin widget: CORS + scoped upload credential + graceful bulk uploads

## Context

C1 from Bug's v0.1.0-rc1 onboarding review: an operator has **no supported way to
authenticate an upload widget hosted on their own site**. Tokens are 15-minute
access JWTs minted only at the admin-vhost `/api/v1/auth/login`, stored in
admin-origin `localStorage` the widget can't read; there is no API-key concept
and no first-class CORS (`OPERATOR_CHECKLIST.md` defers it). Separately, bulk
uploads 503 because the widget fires near-unbounded tus requests past nginx's
`burst=20` (`uploads.go:302` maps storage `ErrServerBusy`ŌåÆ503 `server_busy`).

M0.3 delivers a config-driven coordinator **CORS** layer, a **scoped, revocable
upload credential**, and **graceful bulk uploads** (client queue + server caps).

## Decisions (ratified with Bug, 2026-06-13)

- **One credential primitive, both uses.** A scoped, revocable upload token,
  usable embedded in widget JS (scope it tight) or held server-side (scope
  broad). A browser-embedded token is **public-ish**: CORS constrains *browsers*,
  not `curl`; mitigate by scope + expiry + revocation + size limit + collection
  binding. **Rolling abuse quotas are deferred** to a later feature
  (`upload_credentials.quota.*`), NOT part of this milestone.
- **Three orthogonal concerns, three knobs** (do not collapse): concurrency
  (queue to avoid 503), batch cardinality (the 100-per-session ceiling), abuse
  quota (deferred).
- **`max_files_per_session` server backstop = active in-progress (non-expired)
  tus-session ceiling per credential** (not a rolling window). The widget
  (Uppy `maxNumberOfFiles`) is the primary enforcer; the server guards against a
  client that ignores it.
- **Owner ŌēĀ credential.** `upload_sessions.owner_id` stays a real `users` FK
  (the minting operator). A new nullable `upload_token_id` carries credential
  attribution + the counting key. `auth.Identity` gains `CredentialID`; the
  verifier sets `UserID = token.created_by` (real user), `Role = "uploader"`,
  `CredentialID = token.id` ŌĆö never a synthetic user id.
- **Reuse `RequireRole("uploader",ŌĆ”)`** ŌĆö no upload-route auth change.
- **New caps return 429 + `Retry-After`** (distinct from storage `ErrServerBusy`
  ŌåÆ 503), so the client distinguishes "slow down" from "node overloaded."
- **CORS default-off** (same-origin); opt-in via an origin allowlist.

## Components

### 1. CORS middleware
`internal/api/middleware/cors.go`, applied to the public upload API routes
(`/api/v1/uploads*`, finalize). Config block (Bug's shape):

```yaml
uploads:
  cors:
    enabled: true
    allowed_origins: ["https://example.com"]
    allowed_methods: [GET, POST, PATCH, HEAD, DELETE, OPTIONS]
    allowed_headers: [Authorization, Content-Type, Tus-Resumable, Upload-Length, Upload-Offset, Upload-Metadata]
    exposed_headers: [Location, Upload-Offset, Tus-Resumable, X-Nova-Cid, X-Nova-Envelope-Version]
    allow_credentials: false
```

Preflight `OPTIONS` ŌåÆ 204. `allow_credentials:false` (Bearer, no cookies) ŌćÆ
`Access-Control-Allow-Origin` **echoes** an allowlisted origin + `Vary: Origin`
(not `*`). Disabled (empty allowlist) ŌćÆ no CORS headers (same-origin only).

### 2. Scoped, revocable upload credential
- **Migration `0010_upload_tokens.sql`:**
  - `CREATE TABLE upload_tokens (id uuid pk, token_hash text NOT NULL, label
    text, role user_role NOT NULL DEFAULT 'uploader', collection_id uuid
    REFERENCES collections(id), product blob_product, max_file_size bigint,
    expires_at timestamptz, created_by uuid NOT NULL REFERENCES users(id),
    created_at timestamptz NOT NULL DEFAULT now(), last_used_at timestamptz,
    revoked_at timestamptz)`.
  - `ALTER TABLE upload_sessions ADD COLUMN upload_token_id uuid REFERENCES
    upload_tokens(id)` + partial index `(upload_token_id) WHERE state =
    'in_progress'` for the ceiling query.
- **Wire format:** `nova_ut_<id_b32>.<secret_b64url>`. Store a **SHA-256** hash
  of the secret (it is 128-bit+ random, so a slow password KDF is unnecessary)
  and constant-time compare; `id` is the lookup key.
- **`internal/auth/uploadtoken/verifier.go`** implements `auth.Verifier`:
  recognizes the `nova_ut_` prefix ŌåÆ parse id+secret ŌåÆ lookup by id ŌåÆ
  constant-time hash compare ŌåÆ check `revoked_at IS NULL` and (`expires_at` null
  or future) ŌåÆ `Identity{UserID: created_by, Role: role, CredentialID: id,
  Issuer:"nova:upload-token"}`; best-effort `last_used_at` touch. Non-matching
  prefix ŌåÆ `ErrTokenNotForMe` (chains to the local issuer). Wired into
  `cfg.Verifiers`.
- **Scope enforcement (upload handler):** if `collection_id` bound ŌåÆ target must
  match (else 403); if `product` bound ŌåÆ must match; if `max_file_size` bound ŌåÆ
  tighten the effective limit; persist `upload_token_id` on the session.
- **Admin API (operator-only, admin vhost):** `POST /api/v1/admin/upload-tokens`
  (mint; secret returned **once**), `GET` (list metadata, no secrets), revoke
  (`DELETE` / `POST /{id}/revoke` ŌåÆ set `revoked_at`). + sqlc queries.
- **CLI:** `novactl upload-token create|list|revoke` (M0.3 usability; SPA UI ŌåÆ
  M0.6).
- **`auth.Identity`** gains `CredentialID string` (empty for JWT logins).

### 3. Client-side queue (widget)
- `buildTusOptions`: `limit: cfg.concurrency` (default 4 = `max_concurrent_per_session`)
  ŌćÆ Uppy `autoProceed` keeps a deep queue but only N upload at once ŌåÆ no 503.
- `buildUppy`: `restrictions.maxNumberOfFiles = cfg.maxFilesPerSession`
  (default 100); the `restriction-failed` path surfaces a clear "max N files per
  batch" message.
- `config.ts`: add `concurrency`, `maxFilesPerSession` mount options + `data-*`
  attrs (non-secret).
- `errors.ts`/uploader: map **429** (`too_many_concurrent`) + **503**
  (`server_busy`) to friendly retry messages; tus auto-retries honoring
  `Retry-After`.

### 4. Server caps (backstop)
- `config.Uploads.Limits{MaxConcurrentGlobal:16, MaxConcurrentPerSession:4,
  MaxFilesPerSession:100}` + defaults in `applyUploadDefaults`.
- Admission control in `UploadHandler`:
  - **global** counting semaphore (`MaxConcurrentGlobal`) around in-flight upload
    data ops; full ŌåÆ 429 `too_many_concurrent` + `Retry-After`.
  - **per-credential** (`CredentialID` else `UserID`) concurrency cap
    (`MaxConcurrentPerSession`) ŌåÆ 429.
  - **`CreateTus`**: active in-progress, non-expired sessions for the credential
    `>= MaxFilesPerSession` ŌåÆ 429 (the file-cap backstop; sqlc count query).
- Admin-tunable later (M0.4 config backend + M0.6 modal); M0.3 = operator.yaml +
  defaults.

## Security notes
A browser-embedded upload token is extractable (view-source / `curl`); CORS only
constrains browser origins. Blast radius is bounded by least-privilege scope
(`role=uploader`, optional collection/product binding, size cap), expiry, and
one-click revocation. Admin tokens are **never** minted with operator/moderator
role ŌĆö capped to `uploader`. Rolling per-credential quotas are a deliberate
follow-on (`upload_credentials.quota.max_uploads_per_window`), not M0.3.
