# M6.1 — Master-key file/secret-mount support

## Purpose

Close a v3.1 spec gap surfaced during M6: the master key — the KEK that wraps
every per-blob DEK — can currently *only* be supplied as a plaintext process
environment variable. `internal/envelope/keystore.go` `NewKeystoreFromEnv` reads
`NOVA_MASTER_KEY_<LABEL>` via raw `os.Getenv`, with no `_FILE` / `/run/secrets`
support. This contradicts `REVIEW_2026_05_25.md` C3 ("env OR file-mount;
`NOVA_FOO → NOVA_FOO_FILE → /run/secrets/foo` resolver chain") and
`THREAT_MODEL.md` boundary ③. The inversion is the point: the *less* critical
OIDC signing key (M6) already uses `config.ResolveSecret`; the *most* critical
master key does not. Env-injected secrets leak via `/proc/<pid>/environ`,
`docker inspect`, and APM/crash-dump env scraping, and can persist to disk;
`/run/secrets` is a tmpfs mount that stays in RAM.

## Scope

One function changed: `NewKeystoreFromEnv` in `internal/envelope/keystore.go`.
Its **signature is unchanged** (`(pool) (*Keystore, error)`), so no caller
churn. Everything downstream (`Bootstrap`, `Wrap`, `Unwrap`, the `masters` map,
the `master_key_versions` table) is untouched.

Out of scope: convention directory-scan of `/run/secrets/master-key-*` for
auto-discovering non-active pure-mount labels (deferred to **M10** master-key
rotation if wanted — today there is a single active key, so it adds no operator
UX). Multi-version rotation tooling remains M10.

## Approach (B — resolver chain per label, env-declared discovery)

Replace the env-only scan with:

1. **Build the candidate label set:** `NOVA_MASTER_KEY_ACTIVE` (always) ∪ every
   `NOVA_MASTER_KEY_<LABEL>` (inline) ∪ every `NOVA_MASTER_KEY_<LABEL>_FILE`
   (mount-path declaration) found in `os.Environ()`. Pseudo-labels `ACTIVE` and
   `FILE` are excluded; a `<LABEL>_FILE` var contributes label `<LABEL>`.
2. **Resolve each label** through the shared precedence:
   `config.ResolveSecret("NOVA_MASTER_KEY_<LABEL>", "NOVA_MASTER_KEY_<LABEL>_FILE",
   filepath.Join(defaultSecretsDir, "master-key-"+strings.ToLower(label)))`.
   The mount-path leaf is **lowercased** (`master-key-v1`, not `-V1`) since Linux
   paths are case-sensitive.
3. **Validate:** hex-decode, require `KeySize` (32) bytes. Store under the
   lowercased label.

Because the **active** label is always resolved through the full chain
(including the default mount path), the realistic deployment — drop
`/run/secrets/master-key-v1`, set `NOVA_MASTER_KEY_ACTIVE=v1`, nothing else —
works with no key material in the environment. Additional labels are declared by
an inline value or a `_FILE` env (whose value may itself be a mount).

`defaultSecretsDir` is an unexported package var (`= "/run/secrets"`) so tests
can point it at a `t.TempDir()`. It is global mutable state: the secrets test
file must **not** use `t.Parallel()` and must restore it via `t.Cleanup`.

## Error handling

- **Active label resolves from no source** → refuse to start (the existing
  "ACTIVE=x but … not set" error, reworded to name env / `_FILE` / mount path).
- **Any *declared* label fails to resolve** (e.g. a set-but-unreadable `_FILE`)
  → refuse to start. Never silently skip a declared key — a dropped master-key
  version would render its wrapped blobs unreadable, and silent fallback on an
  explicit secret mapping is a security anti-pattern.
- **Bad hex / wrong length** → existing per-label errors (kept).

## Testing (`internal/envelope/keystore_secrets_test.go`, no `t.Parallel`)

`t.Setenv` + `t.TempDir()` + a `defaultSecretsDir` swap restored via `t.Cleanup`:

1. active via inline env (regression — unchanged behavior).
2. active via `NOVA_MASTER_KEY_V1_FILE=<tmpfile>`.
3. active via the default mount path (`<tmpdir>/master-key-v1`, with
   `defaultSecretsDir` pointed at `<tmpdir>`) and ACTIVE set, **no key env**.
4. multi-label: `v1` inline + `v2` via `_FILE`; both load.
5. errors: bad hex; wrong length; set-but-unreadable `_FILE`; ACTIVE set with no
   resolvable source → each returns an error (no panic).

`NewKeystoreFromEnv` constructs the keystore without touching the DB, so these
tests need no Postgres/Docker.
