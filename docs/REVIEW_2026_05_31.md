# M6.2 Audit-Remediation Review Summary (2026-05-31 → 2026-06-01)

This document captures what changed in M6.2 and why. Read together
with `docs/REVIEW_2026_05_09.md` (v2), `docs/REVIEW_2026_05_19.md`
(v3), and `docs/REVIEW_2026_05_25.md` (v3.1) for the lineage. M6.2
is the first milestone after M6.1 and the first one driven by an
adversarial audit of the as-shipped code rather than a forward-
looking design pass.

## Summary

M1–M6.1 shipped six tagged milestones whose code was broadly sound on
the happy path, but the persistent docs and the working code had
drifted in a handful of places, and the M6 audit surfaced a small set
of real-but-low-blast-radius hardening gaps. M6.2 covers three
classes of work under one branch
(`m6.2-audit-remediation`):

1. **Doc/spec drift fixes** — eight targeted edits to bring
   persistent specs (`README.md`, `docs/ROADMAP.md`,
   `docs/THREAT_MODEL.md`, `docs/specs/ENCRYPTION_ENVELOPE.md`,
   `docs/specs/PRODUCT_MODULE_INTERFACE.md`,
   `docs/specs/SIGNED_URL_FORMAT.md`,
   `docs/superpowers/specs/phase1/2026-05-25-phase1-single-node-mvp-design.md`,
   `internal/auth/localissuer/issuer.go`) back in line with the
   shipped code as of M6.1, plus a new "Try Nova (developer setup)"
   section in the README so curious devs can boot a single-node
   coordinator without the setup wizard (deferred to M13).
2. **Verified code hardening** — seven security fixes (B1–B7), one
   performance fix (C1), and two UX fixes (D1, D2). All landed as
   individual commits, all backed by tests.
3. **Phase 1 delivery-status reconciliation** — ROADMAP now carries
   an honest "Phase 1 — Progress" checklist showing M1–M6.2 tagged
   and a "Phase 1 — Deferred / Future-milestone slots" table
   pointing at the canonical M7–M14 slot list already specified in
   `single-node-mvp-design.md` § "Walking-skeleton milestone
   breakdown". No invented milestone slots.

The full implementation plan lives at
`/home/archbug/.claude/plans/elegant-prancing-squirrel.md` (local-
only, not committed).

### B. Security hardening (verified, code-level)

| # | Finding | Resolution |
|---|---|---|
| B1 | `internal/ratelimit/bucket.go` per-key map was unbounded. The TODO at line 28-32 acknowledged this; with adversarial XFF (B2 was not yet in) or simply many IPv6 sources, the map could grow until OOM. | Added `Config.MaxKeys` (default 100_000) + LRU eviction at the cap + a `Sweep(maxAge)` method. The coordinator's gcLoop sweeps both the public limiter and the login limiter on each tick. Tests cover eviction, sweep, and no-op edge cases. |
| B2 | `middleware/ratelimit.go` and `handlers/upload.go` both consumed the leftmost X-Forwarded-For hop with no trust check. A coordinator reachable directly (no nginx in front) could be rate-limit-bypassed and source-IP-forged by spoofing XFF. | New `NOVA_TRUSTED_PROXIES` env var (comma-separated IPs/CIDRs). One canonical `httputil.ClientIP` helper replaces both ad-hoc implementations: when `RemoteAddr` is in a trusted prefix, the leftmost XFF hop is honored; otherwise XFF is ignored. Empty/unset means XFF is always ignored ⇒ direct-exposure deployments are safe-by-default. Tests cover both topologies. |
| B3 | `internal/auth/localissuer/issuer.go` emitted three distinct `slog.Warn` lines for the user-not-found / disabled / wrong-password login-failure paths, two of which carried `user_id`. Observers of aggregated logs could enumerate which usernames existed and which were disabled. | Collapsed to one constant `slog.Warn("login failed", "request_id", reqID)` line with no per-user differentiator. Granular reason + user_id moved to a parallel `slog.Debug` line so operators at debug log level still get triage detail. Tests verify both levels' contracts via a captured default-logger helper. |
| B4 | `internal/auth/localissuer/refresh.go` reuse-detection path silently swallowed `RevokeRefreshTokenFamily` errors: returned `errRefreshInvalid` → 401 while the family stayed live in the DB. Parallel attackers with sibling tokens kept refreshing successfully. | Added `errRefreshInternal` sentinel and a `retryUntil` helper (2 attempts, 100 ms gap) wrapping the family revoke. Handler maps `errRefreshInternal` to 503 + Retry-After:2. Other "DB error during a critical op" sites (initial hash lookup, child token insert, MarkRefreshTokenRotated) also propagate as `errRefreshInternal` instead of leaking raw errors. Unit tests cover the retry helper directly. |
| B5 | `internal/envelope/keystore.go` and `cmd/coordinator/main.go` silently resolved secrets; an operator who set both inline-env and a `_FILE` redirect had no signal telling them which one actually loaded. | `config.ResolveSecret` now returns `(value, SecretSource, error)`. Keystore emits one `slog.Info` per resolved label carrying `label`, `source` (`env`/`file_env`/`mount`), and `active` flag. Same treatment for the OIDC signing key. No secret bytes are ever logged. Tests verify the per-source contract and that key bytes never appear in captured log output. |
| B6 | `Keystore.Unwrap` called `loadVersions(context.Background())` on a cache miss; during graceful shutdown, a late request could hang on a Postgres query untethered from the request lifetime. | Threaded ctx through `Unwrap(ctx, wrapped, versionID)` and `loadVersions(ctx)`. Single production caller (`storage.OpenBytes`) already had ctx in scope. Integration test asserts the cancellation contract: cancelled-ctx + cache-missing versionID → prompt ctx-error return. |
| B7 | `handlers/upload.go` Multipart trusted `header.Size` for the size cap but Go's multipart parser had already spilled the file part to a temp file on disk before the handler got a chance to reject it. An attacker streaming a small declared file with a huge body could exhaust disk. | `r.Body` wrapped with `http.MaxBytesReader(maxUploadSize + 1 MiB slack)` BEFORE `ParseMultipartForm`. `*http.MaxBytesError` mapped via `errors.As` to 413. File handed to `Put` additionally wrapped with `io.LimitReader` as defense-in-depth (no-op against today's `io.ReadFull(declaredSize)` caller; future-proofing for read-until-EOF callers). Tests cover both the declared-size path and the raw-body overflow path. |

### C. Performance (verified)

| # | Finding | Resolution |
|---|---|---|
| C1 | `DeleteExpiredRefreshTokens` query was `DELETE FROM refresh_tokens WHERE expires_at < now()`. The 0006 schema had a partial index `refresh_tokens_gc_idx` filtering on `WHERE revoked_at IS NULL`, but the query didn't carry that filter so the planner fell back to a sequential scan. Silent until the table grew large. | Split into two queries — `DeleteExpiredRefreshTokens` now reads `expires_at < now() AND revoked_at IS NULL` (matches the existing partial index), and a new `DeleteRevokedRefreshTokensOlderThan($cutoff)` handles explicitly-revoked rows past a 30-day grace. New migration 0007 adds `refresh_tokens_revoked_gc_idx` partial index for the second query. Coordinator's gcLoop calls both. Test seeds five rows across the meaningful (expired/fresh) × (active/revoked-recent/revoked-old) cells and asserts each query deletes exactly its predicate set. |

### D. UX (verified, small-scope)

| # | Finding | Resolution |
|---|---|---|
| D1 | `/health` was the only liveness signal; orchestrators couldn't distinguish "process up" from "all deps reachable". | New `handlers.ReadyHandler` mounted at `/readyz` runs each registered check in parallel under a 1 s deadline. Coordinator wires DB ping + IPFS backend `Health` + (duck-typed) OIDC verifier readiness. JSON body carries per-check name + ok + latency_ms + err. 200 only when every check passes, 503 otherwise. Added `Health(ctx) error` to `ipfs.Backend` interface and `Ready() bool` to `oidc.Verifier`. Two test-fake backends updated to satisfy the new method. |
| D2 | Coordinator startup logged one terse `coordinator: listening on :9000` line via `fmt.Fprintf`; nothing about auth mode, key labels, paranoid flag, etc. | Replaced with one structured `slog.Info("coordinator startup", ...)` line carrying mode, issuer URL, verifier_count, active master-key label, kubo_repo, listen, version, public_uploads, paranoid, trusted_proxies. No secret values. |

The M6.2 changes are additive to the v3.1 spec set:

- `README.md`, `docs/ROADMAP.md`, `docs/THREAT_MODEL.md` brought
  back in line with each other and with the shipped code as of M6.1.

## What did not change

- The trust model — donor-blind, not operator-blind.
- The v1 single-shot envelope codec and its CID-of-the-envelope
  determinism.
- The Phase 2 streaming-AEAD pairing (T1.7a; unchanged from v3.1).
- The single-coordinator topology and the out-of-scope-ness of
  multi-master HA.
- The Nebula + mTLS federation authentication model (Phase 2).
- The five-state liveness model and Tier 1 strict-priority healing.
- The quarantine-first DMCA flow and the legal-hold preservation
  mechanism.
- The M7–M14 slot ordering from the design doc.
- The license (Apache-2.0).

## What still needs deliberation before M7

Nothing blocking. The M7 signed-URL implementation can begin against
the post-A4 `SIGNED_URL_FORMAT.md` and the existing `signing_keys`
schema (present since M1). The /readyz handler added in D1 already
takes a `ReadyCheck` list, so M7 can extend the list with a
"signing_keys" probe if a future operator-facing health check needs
it.
